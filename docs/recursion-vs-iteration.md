# Рекурсія проти ітерації

Якщо ви прийшли з імперативного світу — Java, Python, Go, C++ — то у вашому інструменті "повторити дію кілька разів" завжди є один очевидний кандидат: цикл. `for`, `while`, `do-while` — ці конструкції настільки в'їлися в моторну пам'ять, що мозок навіть не запитує "а як інакше?". Просто `for i in range(n): ...` і поїхали.

У функціональному світі циклів немає. Зовсім. У OCaml `while` існує лише як рудимент для роботи з мутабельними масивами, у Haskell його немає взагалі, у чистих діалектах Lisp його теж немає. Натомість є **рекурсія** — функція, яка викликає сама себе, передаючи трохи зменшену версію вхідних даних.

Цей документ — про те, як перейти з циклового мислення на рекурсивне. Не як на "трюк, який можна використати у задачі N", а як на **основний інструмент структурної декомпозиції**. Ми пройдемо через класичні приклади (сума списку трьома способами), подивимось, як рекурсія працює в самому evaluator-і Swarmlet, обговоримо tail-recursion і чому Python-у це не світить, і закінчимо переліком пасток, у які ви точно потрапите хоча б раз.

Перш ніж занурюватися — варто прочитати поряд [expression-evaluator-explained.md](expression-evaluator-explained.md), бо ми будемо на нього посилатися. А якщо хочеться спочатку побачити загальну картину FP-мислення, гляньте [fp-mental-model.md](fp-mental-model.md). Якщо вас цікавлять алгебраїчні типи даних, які природно живуть у рекурсивній структурі, — [algebraic-data-types.md](algebraic-data-types.md).

## Що насправді робить цикл

Спочатку забудьмо про синтаксис. Що саме робить `for` чи `while`? Ось простий цикл:

```python
total = 0
for x in [1, 2, 3, 4, 5]:
    total = total + x
```

Уважно подивіться на це. Тут є **дві окремі речі**, які ми поєднали в одну конструкцію:

1. **Обхід** — ми проходимо по списку елементів, по одному за раз
2. **Накопичення** — ми тримаємо змінну `total`, яку модифікуємо на кожному кроці

Цикл змішує ці дві ідеї. Ми не описуємо обхід як такий — ми описуємо **дії на кожному кроці** і неявно покладаємося на цикл, що він їх повторить. Стан накопичення схований у мутабельній змінній `total`, яка живе **поза** тілом циклу. Для імперативного програміста це настільки природно, що навіть не помічається.

У функціональному стилі ми розділяємо ці речі. Обхід — це властивість **структури даних**, а не керування потоком. Список — це не "набір елементів, по яким можна пройти у циклі". Список — це **рекурсивна структура**: він є або порожнім (`[]`), або складається з голови і хвоста (`x :: xs`). Накопичення — це окрема функція, яку ми застосовуємо до елементів. Замість "пройди по списку, кожного разу додай" ми кажемо "обчислити суму списку — це або 0 (якщо список порожній), або голова плюс сума хвоста".

Це інший спосіб думати. Ви описуєте **що означає сума списку**, а не **як її обчислити крок за кроком**. Якщо це звучить надто абстрактно — нічого, далі буде код.

## Сума списку трьома способами

Класична задача. Маємо список чисел, треба отримати їх суму.

### Версія 1: Python, ітеративна

Ось як це пише імперативний інженер:

```python
def sum_list(xs):
    total = 0
    for x in xs:
        total = total + x
    return total
```

Все знайомо. Ініціалізуємо акумулятор, проходимо по списку, на кожній ітерації мутуємо акумулятор. Повертаємо результат. Це **переказ алгоритму крок за кроком**.

Зверніть увагу на те, що тут є **два рівні мутації**: `total = total + x` (явна мутація змінної) і неявна "поточна позиція в списку", якою керує сам `for`. Друга мутація схована, але вона є — Python тримає внутрішній ітератор і пересуває його на кожній ітерації.

### Версія 2: Python, рекурсивна

А ось як це можна написати без циклу:

```python
def sum_list(xs):
    if not xs:
        return 0
    return xs[0] + sum_list(xs[1:])
```

Прочитайте це уважно. Тут немає мутації. Немає `total`. Немає `for`. Є дві фрази:

- "Якщо список порожній — сума дорівнює 0."
- "Інакше — сума дорівнює першому елементу плюс сума решти списку."

Це **визначення**, а не алгоритм. Воно описує, **що таке** сума списку, а не **як її обчислити**. Обчислення — це наслідок визначення, а не основна суть.

Зверніть увагу на дві речі. По-перше, є **базовий випадок** — `if not xs`. Без нього рекурсія йшла б до нескінченності. По-друге, рекурсивний виклик передає **меншу** версію проблеми — `xs[1:]` коротше за `xs`. Без цього теж була б нескінченна рекурсія.

Це загальний шаблон будь-якої коректної рекурсії: **базовий випадок + крок до базового випадку**. Якщо у вашій рекурсивній функції немає одного з цих елементів — вона зламана. Це аналог `i < n` і `i = i + 1` у класичному `for` — ви маєте умову зупинки і рух у бік цієї умови.

### Версія 3: OCaml, рекурсивна

Ось як це виглядає у мові, де рекурсія — основний спосіб робити речі:

```ocaml
let rec sum_list xs =
  match xs with
  | [] -> 0
  | x :: rest -> x + sum_list rest
```

Це майже точно те саме, що Python-варіант, але з трьома відмінностями.

По-перше, `let rec` замість `def`. `rec` каже компілятору "ця функція може посилатися на себе". OCaml без `rec` не дозволяє рекурсивних викликів — це робиться явно, щоб не було випадкових циклів.

По-друге, `match xs with` замість `if`. Замість того щоб питати "чи список порожній", ми **розкладаємо структуру** і обробляємо обидва випадки одразу. `[]` — це порожній список, `x :: rest` — це список з головою `x` і хвостом `rest`. Pattern matching робить структуру списку видимою у синтаксисі.

По-третє, синтаксис обходу. У Python ми пишемо `xs[0]` і `xs[1:]` — це індексація і slice, які працюють з будь-якою послідовністю. У OCaml `x :: rest` — це **деструктуризація**, яка працює тільки зі списками і робить це за O(1) часу. У Python `xs[1:]` створює новий список (O(n)), у OCaml `rest` — це просто посилання на існуючий хвіст списку (O(1)).

Це не косметична різниця. У OCaml списки **спроектовані** для рекурсивної обробки — вони immutable single-linked list, де доступ до голови і хвоста за O(1). У Python списки — це масиви, і рекурсивний обхід через slice — антипатерн з точки зору продуктивності. Якщо ви пишете рекурсію над Python-списками, краще передавати індекс, а не slice:

```python
def sum_list(xs, i=0):
    if i >= len(xs):
        return 0
    return xs[i] + sum_list(xs, i + 1)
```

Або взагалі ітератор. Але з точки зору **мислення** — рекурсивна форма ясно показує структуру задачі.

## Рекурсія в evaluator-і Swarmlet

Тепер — найцікавіше. Подивіться на `eval_expr` у [swarmlet/eval.py](../swarmlet/eval.py):

```python
def eval_expr(node, ctx):
    if isinstance(node, A.Num):
        return node.value
    if isinstance(node, A.Bool):
        return node.value
    if isinstance(node, A.Var):
        return _eval_var(node, ctx)
    if isinstance(node, A.BinOp):
        return _eval_binop(node, ctx)
    if isinstance(node, A.If):
        cond = eval_expr(node.cond, ctx)
        if _ensure_bool(cond, node.line):
            return eval_expr(node.then_expr, ctx)
        return eval_expr(node.else_expr, ctx)
    if isinstance(node, A.Let):
        val = eval_expr(node.value, ctx)
        child_ctx = ctx.child(locals={node.name: val})
        return eval_expr(node.body, child_ctx)
    ...
```

Це **класична рекурсія по структурі**. AST — це дерево, де кожен внутрішній вузол містить інші вузли. `eval_expr` обходить це дерево, на кожному вузлі обираючи правильну гілку обробки і **рекурсивно** обчислюючи піддерева.

Подивіться на гілку для `A.If`. Що вона робить?

1. Рекурсивно обчислює `node.cond` (умову — це теж вираз, теж дерево)
2. Перевіряє, чи це bool
3. Якщо `true` — рекурсивно обчислює `then_expr`
4. Якщо `false` — рекурсивно обчислює `else_expr`

Це не "ітерація по умовах". Це **структурна декомпозиція**. `If` як AST-вузол складається з трьох піддерев — `cond`, `then_expr`, `else_expr`. Щоб обчислити `If`, ми обчислюємо ці піддерева (рекурсивно, бо вони такі ж AST-вузли) і поєднуємо результати.

Гілка для `A.Let` — те саме. `let x = e1 in e2` — це AST-вузол з двома піддеревами (`value` і `body`). Щоб його обчислити, ми спочатку обчислюємо `value` (рекурсивно), потім додаємо біндинг до контексту і обчислюємо `body` (теж рекурсивно).

Якби ми писали це ітеративно, нам довелося б **руками** реалізувати стек — пушити вузли, які треба обчислити, попити проміжні результати, поєднувати. Це робиться (і робиться у "справжніх" virtual machines типу JVM-ового JIT), але код стає в десять разів більшим і важчим для розуміння. У вигляді рекурсивної функції `eval_expr` поміщається на одному екрані і **читається як визначення мови**.

Ось чому інтерпретатори майже завжди пишуть рекурсивно. Структура коду відображає структуру даних, які він обробляє. AST — рекурсивна структура (дерево), evaluator — рекурсивна функція. **Form follows form.**

Більше деталей про конкретні гілки `eval_expr` — у [expression-evaluator-explained.md](expression-evaluator-explained.md). Тут нам важливе одне: **сам факт того, що це рекурсія, а не цикл, і чому це природно**.

### Tree traversal як fold

Загальніше: будь-який обхід дерева — це **fold**. Ви визначаєте, як комбінувати результати дітей у результат батька, і fold проходить по дереву знизу вгору, поєднуючи. `eval_expr` — це fold по AST, де "комбінування" — це інтерпретація операторів.

Якщо у вас в голові вкоренилася ідея "fold = згорнути список" — розширте її. Fold працює **на будь-якій рекурсивній структурі**: списках, деревах, графах (з обережністю). Скрізь, де є базовий випадок і композиційний крок, є fold.

## Tail recursion і акумулятори

Є проблема з рекурсією, на яку ми ще не натрапили: **глибина стеку**. Кожен рекурсивний виклик додає кадр у call stack. Якщо рекурсія глибока, стек переповнюється.

Подивіться на нашу `sum_list`:

```python
def sum_list(xs):
    if not xs:
        return 0
    return xs[0] + sum_list(xs[1:])
```

Що відбувається при `sum_list([1, 2, 3])`? Виклик розгортається так:

```
sum_list([1, 2, 3])
  = 1 + sum_list([2, 3])
  =     1 + (2 + sum_list([3]))
  =         1 + (2 + (3 + sum_list([])))
  =             1 + (2 + (3 + 0))
  =                 1 + (2 + 3)
  =                     1 + 5
  =                     6
```

Зверніть увагу — щоб обчислити верхній виклик, нам треба тримати в пам'яті **всі проміжні `+`** до того моменту, поки ми не дійдемо до базового випадку. Стек росте на кожному кроці. На списку з мільйона елементів стек переповниться.

### Акумулятор-патерн

Розв'язок — **переписати рекурсію так, щоб результат викликав себе сам, а не ставав аргументом до подальшого обчислення**. Це називається **tail recursion** — рекурсивний виклик стоїть в "хвостовій" позиції, останнім, його результат — це і є результат функції.

Ось ідея на OCaml:

```ocaml
let sum_list xs =
  let rec aux xs acc =
    match xs with
    | [] -> acc
    | x :: rest -> aux rest (acc + x)
  in
  aux xs 0
```

Тут є зовнішня функція `sum_list` і внутрішня `aux`, яка тягне з собою **акумулятор** `acc`. Замість того щоб "обчислити суму хвоста, потім додати голову", ми "додаємо голову до акумулятора зараз, потім просимо аналогічну функцію обчислити решту з оновленим акумулятором".

Розгортка `sum_list [1; 2; 3]` тепер виглядає інакше:

```
sum_list [1; 2; 3]
  = aux [1; 2; 3] 0
  = aux [2; 3] 1
  = aux [3] 3
  = aux [] 6
  = 6
```

Кожен крок — **повна заміна виклику**. Ні в одному місці нам не треба тримати "недороблене обчислення" з зовнішнього виклику. Це і є tail call.

Компілятор OCaml розпізнає таку структуру і **переписує рекурсію в цикл**. Це називається **tail call optimization (TCO)**. Згенерований машинний код буде ідентичний тому, що дав би `for`-цикл — той самий регістровий accumulator, той самий jump. Стек не росте взагалі. Можна обробити список з мільярда елементів без проблем.

Це не оптимізація "якщо повезе" — у Scheme це **частина мовного стандарту**: tail calls **зобов'язані** не нарощувати стек. У OCaml, F#, Scala (для самовикликів), Haskell — те саме. Ви можете покладатися на це і писати рекурсивні цикли так само сміливо, як імперативні `for`.

## Чому Python не має TCO

Питання, яке виникає в кожного, хто прийшов з ML-світу: **а в Python є TCO?** Відповідь — ні. І ніколи не буде.

Це усвідомлений вибір Гвідо ван Россума, ще з 2009 року. Він написав окремий блог-пост ("Tail Recursion Elimination"), де пояснив свою позицію. Аргументи такі:

**Стек-трейси важливіші за глибокі рекурсії.** Якщо TCO переписує рекурсивні виклики в цикли, у traceback-у пропадає історія викликів. Ви не побачите, який саме виклик призвів до помилки — побачите тільки фінальний кадр. Для відлагодження це біль. Гвідо вважає, що зрозумілий traceback — фундаментальна риса Python, важливіша за можливість писати безкінечно глибоку рекурсію.

**Python — не функціональна мова.** У Python є цикли. Якщо ви хочете щось повторити багато разів — використовуйте `for` чи `while`. Рекурсія в Python є, але як інструмент для **рідкісних задач, де структура природно дерев'яна** — обхід AST, парсинг, файлова система. Для лінійних задач ніхто не вимагає TCO, бо рекурсія тут зайва.

**Стек-ліміт як safety net.** За замовчуванням Python обмежує глибину рекурсії десь на 1000 (`sys.getrecursionlimit()`). Це навмисне — якщо ви випадково написали безкінечну рекурсію, ви отримаєте `RecursionError` через секунду, а не повний звис системи через годину. Якщо потрібна глибша рекурсія — є `sys.setrecursionlimit(10000)`, але це червоний прапорець "перепиши ітеративно".

**Філософська позиція.** "There should be one — and preferably only one — obvious way to do it." В Python обвиний спосіб циклу — `for`. Не два способи (рекурсія + цикл), один. Гвідо не хоче, щоб Python ставав другорядною Lisp.

Чи можна обійти? Так, є хаки — батути (trampolines), декоратори, що ловлять рекурсію через виняток. Існують пакети типу `tco` чи `tailrec`. Але вони повільні, ламають traceback, і ніхто їх в реальному коді не використовує. Якщо вам **дуже** потрібна глибока рекурсія в Python — перепишіть як цикл або як while + явний стек. Це сумно для людини з функціональним смаком, але це реальність екосистеми.

## Що це означає для evaluator-у Swarmlet

Якщо Python не має TCO — а evaluator Swarmlet написаний в Python і **рекурсивний**, — чи не вибухне він на складній програмі?

Коротка відповідь: **ні, на практиці.** Довша відповідь — нижче.

Глибина рекурсії `eval_expr` дорівнює **глибині AST**. Кожен рекурсивний виклик `eval_expr` — це один рівень вкладеності виразу. Тобто на скільки глибоко ви можете занести `let ... in (let ... in (let ... in ...))` чи арифметичних виразів — настільки буде стек.

У Swarmlet AST глибина обмежена кількома факторами:

- **Програми Swarmlet — короткі.** Декларативна мова для cellular automata. Типова cell rule — 5-15 рядків. AST такого правила має глибину 10-30 вузлів. Це навіть близько не межі стеку Python.
- **Граматика обмежена.** Немає вкладених функцій, немає замикань, немає динамічних модулів. Все, що є — арифметика, `if`, `let`, `match`, виклики builtin-ів. Складність обмежена синтаксисом.
- **Парсер відкине надто складне.** Якби хтось згенерував AST глибиною 1000, парсер на нього сам впав би (recursive descent parser). Це означає, що до evaluator-у такий AST не дійде.

Інакше кажучи, ми **знаємо**, що глибина AST обмежена розумним числом. Стек у 1000 — це у 30+ разів більше за реалістичну глибину. Можемо спокійно покладатися на нативну рекурсію Python без TCO.

Якби Swarmlet мав general-purpose обчислення (типу loop-and-recur), цього було б недостатньо. Тоді довелося б або переписувати evaluator як ітеративний state machine з explicit stack, або ставити harсh-ліміт на глибину виразів. Але оскільки Swarmlet — DSL для grid-based simulation, така інженерія не потрібна. **Рекурсія тут "просто працює", бо домен її не напружує.**

## Map, filter, fold як рекурсивні патерни

Якщо у вас в голові вже сидить ідея "рекурсія = базовий випадок + крок", то ви зрозумієте, чому **map, filter, fold** — це не три окремих функції, а три варіанти **одного й того ж рекурсивного шаблону**.

**Map** — застосувати функцію до кожного елемента:

```ocaml
let rec map f xs =
  match xs with
  | [] -> []
  | x :: rest -> (f x) :: map f rest
```

Базовий випадок: порожній список → порожній список. Крок: застосуй `f` до голови, рекурсивно map-ни хвіст.

**Filter** — залишити тільки елементи, які задовольняють предикат:

```ocaml
let rec filter pred xs =
  match xs with
  | [] -> []
  | x :: rest ->
    if pred x then x :: filter pred rest
    else filter pred rest
```

Базовий випадок: порожній → порожній. Крок: якщо голова проходить — пиши її і фільтруй хвіст; якщо ні — просто фільтруй хвіст.

**Fold** — узагальнення обох:

```ocaml
let rec fold_left f acc xs =
  match xs with
  | [] -> acc
  | x :: rest -> fold_left f (f acc x) rest
```

Базовий випадок: порожній → акумулятор. Крок: оновити акумулятор через `f`, рекурсивно згорнути хвіст.

Зверніть увагу — `fold_left` написана у tail-recursion стилі (з акумулятором), а `map` і `filter` — у "звичайному" стилі. Можна було б переписати їх через fold:

```ocaml
let map f xs = List.fold_right (fun x acc -> f x :: acc) xs []
let filter pred xs = List.fold_right (fun x acc -> if pred x then x :: acc else acc) xs []
```

Це показує **fundamental insight функціонального програмування**: майже все, що ви робите зі списком, — це різновид fold. Ітерація по списку — це fold з порожнім акумулятором. Підрахунок елементів — fold, який ігнорує елементи і додає 1. Пошук max — fold, який порівнює і тримає найбільше.

Якщо рекурсивний шаблон стає для вас природним, ви побачите fold у задачах, де раніше бачили цикл. Це не зменшує продуктивність — на компільованих ML-мовах fold-функції inline-ються до того ж машинного коду, що й ручний цикл. А на читабельність — впливає колосально.

## Python-паралель: list comprehensions як fold

Хороша новина для Python-програмістів — у вас **уже є fold-подібний інструмент**. Це list comprehension.

Подивіться:

```python
squares = [x * x for x in xs]              # map
evens = [x for x in xs if x % 2 == 0]      # filter
total = sum(x for x in xs)                 # fold (через sum)
```

Перший рядок — `map`. Другий — `filter`. Третій — fold через builtin `sum`. Жоден з них не використовує явний цикл з мутацією — все це **декларативні описи** того, що ви хочете отримати.

`reduce` з `functools` — це загальний fold:

```python
from functools import reduce
total = reduce(lambda acc, x: acc + x, xs, 0)
product = reduce(lambda acc, x: acc * x, xs, 1)
```

Pythonic-style надає перевагу `sum()`, `min()`, `max()`, `any()`, `all()` для типових fold-ів, а `reduce` залишається для нестандартних. Гвідо взагалі хотів викинути `reduce` з мови (і викинув з builtins у Python 3), але спільнота протестувала, тому він живе у `functools`.

Comprehensions — це не **просто синтаксичний сахар** для циклів. Це **декларативний інтерфейс** до того ж рекурсивного шаблону, що й fold у ML-мовах. Коли ви пишете `[f(x) for x in xs]`, ви не описуєте "пройди в циклі" — ви описуєте "результат — це список значень `f(x)` для кожного `x` з `xs`". Це **визначення множини**, не алгоритм.

У внутрішньому коді Python comprehension компілюється у байт-код циклу — так само як OCaml-овий `List.map` компілюється у tail-call машинний код. Але на рівні **думки** comprehension і map — одне й те саме.

## Коли в Python використовувати рекурсію, а коли цикл

Раз TCO немає, а традиція мови — циклова, виникає питання: **коли все ж писати рекурсію в Python?**

Загальне правило: **рекурсія, коли структура даних рекурсивна. Цикл — інакше.**

Конкретніше:

**Пишіть рекурсивно**, якщо:
- Ви обходите дерево (AST, файлова система, JSON, HTML DOM, BST)
- Ви обходите граф depth-first (з відстеженням відвіданих)
- Ви парсите за recursive descent
- Структура задачі природно ділиться на "базовий випадок + менша підзадача" (binary search, divide-and-conquer типу merge sort)
- Ви обчислюєте щось на основі визначення, яке саме рекурсивне (factorial, fibonacci — хоча для них краще ітеративні версії)

**Пишіть цикл**, якщо:
- Ви проходите лінійну структуру (список, range, файл по рядках)
- Ви накопичуєте щось у мутабельну колекцію (`list.append` в циклі)
- Глибина потенційно велика (більше за кілька сотень)
- Ви хочете легкого `break`, `continue`, ранніх виходів

Гібрид — теж нормально. Внутрішня логіка може бути рекурсивною (обхід дерева), а зовнішня — циклова (для кожного дерева в списку). Не намагайтеся бути ML-ним за всяку ціну. Python — це Python.

## Поширені пастки

Чотири помилки, у які потрапляють всі, хто вчиться писати рекурсивно. Перевірте, чи не робите ви їх.

**Пастка 1: відсутній або неправильний базовий випадок.** Класика — забути перевірку на порожній вхід. `def length(xs): return 1 + length(xs[1:])` — це нескінченна рекурсія. Або базовий випадок є, але неправильний — `if xs == [0]: return 0`, а потім хтось викликав на `[]`, і ви впали. Завжди починайте з базового випадку. Перш ніж писати рекурсивний крок — напишіть, що повертається для "найменшого" входу.

**Пастка 2: рекурсивний крок не наближається до базового випадку.** `def f(xs): return f(xs)` (передаємо те саме). Або `def f(n): return f(n - 0)` (зменшуємо, але на нуль). Перевірте, що **кожен** рекурсивний виклик отримує "ближчі" дані. Якщо обхід списку — `xs[1:]`, не `xs`. Якщо обхід числа — `n - 1`, не `n`. Якщо обхід дерева — діти, не корінь.

**Пастка 3: повторні обчислення без мемоїзації.** Наївна рекурсивна `fibonacci`:

```python
def fib(n):
    if n < 2:
        return n
    return fib(n - 1) + fib(n - 2)
```

`fib(40)` буде рахуватися годинами, бо ті самі піддерева обчислюються мільйони разів. Розв'язок — `@functools.lru_cache` декоратор, або переписати ітеративно. Це не помилка рекурсії як такої — це помилка **наївної рекурсії без memoization**. У ML-світі цього менше, бо там часто пишуть з акумулятором.

**Пастка 4: рекурсія там, де треба ітерація.** Ви прочитали цей документ, надихнулися, і починаєте писати все рекурсивно. `def print_all(xs): print(xs[0]); print_all(xs[1:])`. На списку з 2000 рядків ви отримуєте `RecursionError`. Не треба. У Python на лінійних задачах **`for` — правильний інструмент**. Рекурсія — для того, що **природно** рекурсивне.

## Вправи

Чотири задачі, від простого до складного. Розв'яжіть хоча б першу і останню — це покриє більшість шаблонів.

### Вправа 1: довжина списку рекурсивно

**Постановка.** Напишіть функцію `length(xs)` без використання `len()` та без циклів, тільки рекурсивно.

**Завдання.** Зробіть це у двох варіантах: (а) звичайна рекурсія `1 + length(xs[1:])`, (б) tail-recursive з акумулятором.

**Підказка.** Базовий випадок — порожній список має довжину 0. Для tail-recursive передавайте `acc` як другий параметр, починаючи з 0.

### Вправа 2: реверс списку через fold

**Постановка.** Напишіть `reverse(xs)`, що повертає список у зворотному порядку. Через `functools.reduce` (без `xs[::-1]`, без циклу).

**Завдання.** Спочатку напишіть рекурсивну версію (`reverse(rest) + [x]`). Потім перепишіть через `reduce(lambda acc, x: [x] + acc, xs, [])`. Поясніть, чому друга версія не tail-recursive у Python (попри назву `reduce`).

**Підказка.** `[x] + acc` створює новий список кожного разу — це O(n) на кожному кроці. Загальна складність вийде O(n²). Як це виправити? (Підказка: `acc.append(x)` у lambda — і поверніть `acc`. Або просто використовуйте `acc + [x]` навпаки. Подумайте, який порядок дасть правильний результат.)

### Вправа 3: pretty-print AST Swarmlet

**Постановка.** Подивіться на `swarmlet/ast.py`. AST складається з вузлів — `Num`, `Bool`, `Var`, `BinOp`, `If`, `Let`, `Match`, `Call`. Напишіть функцію `pretty(node, indent=0)`, що друкує AST у read-friendly вигляді з відступами.

**Завдання.** На вузлі `BinOp` друкуйте `BinOp(<op>)` і рекурсивно друкуйте `left` і `right` з більшим відступом. На `If` — три гілки. На `Let` — назва і два піддерева. На листках (`Num`, `Bool`, `Var`) — просто значення.

**Підказка.** Це той самий шаблон, що `eval_expr` — `if isinstance(node, A.Num): ...`. Тільки замість обчислення значення ви форматуєте рядок. Рекурсія йде по піддеревах. Не забудьте про `Match` — він має список `cases`, кожен з яких має `body` і опціональний `guard`.

### Вправа 4: обхід файлової системи

**Постановка.** Напишіть функцію `total_size(path)`, що повертає суму розмірів усіх файлів у `path` рекурсивно. Тільки `os.path` і `os.listdir`, без `os.walk`.

**Завдання.** Базовий випадок — це файл (`os.path.isfile`), повертаємо `os.path.getsize`. Рекурсивний випадок — це директорія, повертаємо суму `total_size` для кожного дочірнього елемента.

**Підказка.** Це класичний tree fold: листки — файли, внутрішні вузли — директорії. Думайте про файлову систему як про AST, де `total_size` — це `eval_expr`, що замість значень рахує байти. Якщо файлова система глибока (>1000 рівнів) — отримаєте `RecursionError`, і це нормально, перепишіть через `os.walk` або deque-based BFS. Це гарна нагода відчути межу — рекурсія прекрасна, але не безмежна.

## Підсумок

Якщо коротко: **цикл — це повторення дії, рекурсія — це декомпозиція структури**. Імперативні мови дають вам цикл як основний інструмент і рекурсію як backup. Функціональні мови — навпаки. Перехід між світоглядами — це не вивчення нового синтаксису, а навчання бачити дані як рекурсивні структури і функції як рекурсивні визначення.

У Swarmlet evaluator-і ми бачимо це наочно — `eval_expr` написана як визначення, а не як алгоритм. Це робить її читабельною як специфікацію мови, а не як код. Tail recursion дозволяє писати рекурсивні цикли в ML-мовах без втрат продуктивності, Python її не має, але і не потребує — для рідкісних випадків глибокої рекурсії перепишіть на цикл, для типових випадків декларативного опису — використовуйте comprehensions і `functools.reduce`.

Найголовніше — пам'ятайте про базовий випадок і крок до нього. Це аналог `i < n` і `i++` у `for`-циклі. Без них рекурсія не працює, як і цикл без умови виходу. Все інше — справа практики.
