# Intent pattern: чому агент не діє відразу, а спочатку каже, що збирається зробити

Уявіть собі грід 100 на 100. На цьому гріді бігає 40 вовків і 200 овець. Вони голодні, погано виховані і не знають про чергу. На якомусь тіку відбувається така ситуація: дві клітинки на північ і на південь від однієї овечки стоять два вовки. Обидва бачать овечку через `nearest_agent_of_type_dir Sheep wolf_vision`, обидва вирішують рухатися до неї, обидва доходять, обидва вирішують `kill Sheep`. Що має статися?

Якщо ви програмували імперативно достатньо довго, ви відразу відчуваєте знайому неприємність у животі. Ви знаєте, що буде далі: race condition, undefined behavior, mutex, retry-логіка, можливо ще `optimistic concurrency control`. Ви знаєте, що тести будуть проходити локально і падати на CI, бо порядок виконання інакший. Ви знаєте, що один і той самий seed дасть різні результати, якщо хоча б одна функція стане трохи повільнішою.

У Swarmlet цього всього немає. Той самий seed дає той самий результат байт-у-байт, і це не магія — це наслідок одного дуже простого архітектурного рішення. Агент **не діє безпосередньо**. Агент **описує**, що він хоче зробити, у вигляді запису, який називається `Intent`. Усі інтенти збираються по всіх агентах, і **тільки потім** застосовуються — у фіксованому, документованому порядку.

Це і є intent pattern. Цей документ пояснює, чому без нього не можна було б побудувати детермінований движок, як він реалізований у файлі [`swarmlet/engine.py`](../swarmlet/engine.py), і чим він схожий на половину архітектурних патернів, які ви вже знаєте з імперативного світу — Redux, transactions, event sourcing, Command pattern, CRDT.

Перед тим як занурюватися, рекомендую прочитати [SPEC.md секція 2.3](../specification/Swarmlet-SPEC.md#23-tick) (опис тіка і порядок резолюції конфліктів) і [SPEC.md секція 7.3](../specification/Swarmlet-SPEC.md#73-agent-rule-evaluation) (як саме агентські правила перетворюються на інтенти). Цей документ — комплементарний до них: SPEC говорить **що**, тут пояснюється **чому**.

## Сценарій: два вовки і одна вівця

Давайте конкретизуємо. У `swarmlet/examples/wolf_sheep.swl` правило вовка приблизно таке:

```swl
let agent Wolf =
  seq {
    set energy = self.energy - 1;

    if agents_of_type_in_radius Sheep 0 > 0
      then seq {
             kill Sheep;
             set energy = self.energy + wolf_kill_gain
           }
      else stay;

    let prey_dir   = nearest_agent_of_type_dir Sheep wolf_vision in
    let prey_count = agents_of_type_in_radius Sheep wolf_vision in
    if prey_count > 0
      then move prey_dir
      else move (random_dir ());

    if self.energy > wolf_repro_thresh
      then seq { set energy = wolf_repro_cost; spawn Wolf }
      else stay;

    if self.energy <= 0 then die else stay
  }
```

Один вовк за тік може: змінити свою енергію, вбити вівцю, рухатися, заспавнити нового вовка, померти. П'ять різних типів ефектів на світ. А тепер уявіть, що у вас 40 вовків. У кожному тіку відбувається до 200 окремих ефектів, які потенційно конфліктують один з одним. Дві клітинки куди хочуть пройти двоє вовків — вони обидва не можуть стояти там, або можуть, але треба чітко визначити, хто там опинився першим. Двоє вовків на одній клітинці хочуть вбити одну вівцю — вівця має померти один раз, а не два. Один вовк помирає у тому самому тіку, що і інший спавнить нового — нащо новому народжуватися, якщо батько вже мертвий?

Це і є той самий "two wolves chasing one sheep" сценарій, який в імперативному коді став би мукою. Подивимось, як його робить наївний підхід, і чому це не працює.

## Наївний дизайн і чому він валиться

Перший варіант, який спадає на думку імперативному програмісту, виглядає приблизно так:

```python
def evaluate_agent(agent, world):
    if agent.cell_state == "Grass":
        agent.energy -= 1

    if has_sheep_here(world, agent.x, agent.y):
        kill_one_sheep(world, agent.x, agent.y)
        agent.energy += wolf_kill_gain

    direction = nearest_sheep_dir(world, agent)
    move_agent(world, agent, direction)

    if agent.energy > wolf_repro_thresh:
        spawn_new_wolf(world, agent.x, agent.y)
        agent.energy = wolf_repro_cost

    if agent.energy <= 0:
        kill_agent(world, agent)


def tick(world):
    for agent in world.agents:
        evaluate_agent(agent, world)
```

Виглядає чисто, просто, зрозуміло. І не працює. Розглянемо чому, по пунктах.

**Order-dependence.** Цикл `for agent in world.agents` обходить агентів у якомусь порядку. У Python це порядок вставки у список. Якщо вовк A йде перший і з'їдає вівцю, вовк B бачить порожню клітинку і не отримує бонусу. Якщо ви заведете два потоки і паралельно обходите половину списку — порядок змінюється з кожним запуском. Висновок: результат симуляції залежить від порядку обходу. Один і той самий seed дасть різні snapshot-и при різних реалізаціях ітерації. Детермінізму немає.

**Race conditions без перебільшення.** Поки вовк A читає `nearest_sheep_dir`, цей метод дивиться на положення інших вовків і овець. Але вовк B, обчислений раніше у цьому тіку, вже зрушив з місця. Вовк A сприймає світ, у якому вовк B стоїть на двох клітинках одночасно — на старій (бо A це бачить через старі дані) і на новій (бо B вже зрушив). Світ "розповзається" по тіку. Логіка стає некогерентною.

**Conflict resolution неможлива.** Якщо два вовки одночасно хочуть піти на ту саму клітинку, перший прийде, другий зустріне там вже зайняту клітинку і має виконати якусь fallback-логіку. Але хто з них "перший"? Перший у списку? Це довільно. У хорошому дизайні мав би бути механізм випадкового вибору переможця з рівними шансами. Але якщо переможець залежить від ще ненаписаної fallback-логіки кожного агента, ви маєте локальну логіку, що приймає глобальні рішення про конфлікти — це антипатерн чистої води.

**Snapshot як абстракція ламається.** Чисто функціональна модель Swarmlet каже: правило — це функція з `(snapshot, agent) -> intent`. Якщо ви мутуєте snapshot під час обчислення, ця функція стає `(mutating snapshot, agent) -> ???`, і ви більше не можете гарантувати ні чистоту, ні референційну прозорість, ні передбачуваність. Ви просто пишете звичайний імперативний код, замаскований під DSL.

**Одна функція, кілька ефектів.** Вовк рухається, потім спавнить, потім помирає. Кожен ефект окремо може здатися безпечним, але разом вони створюють короткі стани, у яких світ некогерентний. Вовк уже спавнив дитину, але ще не помер — у списку є і батько, і дитина. Якщо у наступному рядку батько помирає, дитина залишилася, і це OK, але якщо ви читаєте список агентів між цими двома операціями, ви бачите фантомний стан.

Висновок очевидний: треба розв'язати **обчислення ефектів** і **їх застосування**. Це і робить intent pattern.

## Intent pattern в одному реченні

Опиши, що ти хочеш зробити (збери запис), збери всі описи, застосуй атомарно з детермінованою резолюцією конфліктів.

Або довшим, точнішим формулюванням: правило агента — це чиста функція від `(snapshot, agent) -> Intent`, де `Intent` — це **запис всіх ефектів, що агент хоче зробити на світ**. Снапшот не мутується. Список інтентів збирається. Потім окрема функція застосування інтентів пробігає по них **у документованому порядку пріоритетів**, виконуючи рухи, записи у поля, спавни, смерті, з повним контролем над конфліктами.

Це звучить як проста реструктуризація. Насправді це фундаментальна зміна моделі обчислень. Замість "агент діє у світі" — "агент описує дію, окремий механізм світу її виконує". Це **відокремлення опису від виконання**, що є серцем багатьох архітектурних патернів.

### Запис `Intent`

Ось що насправді є у [`swarmlet/engine.py`](../swarmlet/engine.py):

```python
@dataclass
class Intent:
    """Accumulates actions for a single agent during one tick."""
    agent_id: int
    move_dir: int = STAY
    agent_field_writes: Dict[str, float] = field(default_factory=dict)
    cell_field_deposits: List[Tuple[str, float]] = field(default_factory=list)
    spawn_types: List[str] = field(default_factory=list)
    kill_targets: List[str] = field(default_factory=list)
    die: bool = False
```

Розглянемо поля по черзі.

**`agent_id`** — кому належить цей інтент. Один агент — один інтент за тік. Щоб не плутатися, коли застосуємо.

**`move_dir`** — у якому напрямку агент хоче рухатися. `STAY` означає "залишитися на місці". Кодується як `int 0..7` для восьми сторін Moore-сусідства плюс `STAY = -1` (значення сидить у `swarmlet.builtins`).

**`agent_field_writes`** — словник `{field_name: new_value}` записів у власні поля агента. Якщо вовк викликає `set energy = self.energy - 1`, цей запис потрапляє сюди.

**`cell_field_deposits`** — список депозитів у клітинні поля. Чому список, а не словник? Бо депозити **сумуються**, не перезаписуються. Якщо два мурахи сядуть на одну клітинку і обидва задепозитять `pheromone += 0.5`, ефект має бути `+1.0`, а не "хто останній, того тапки". Список зберігає всі депозити окремо, сумування відбувається у фазі застосування.

**`spawn_types`** — список типів агентів, яких треба заспавнити. Знов список, бо за один тік один агент теоретично може заспавнити декількох (хоча на практиці у наших прикладах це один за раз).

**`kill_targets`** — список типів агентів, яких треба вбити на цій клітинці. Кожен елемент — це окремий інтент "вбий одного агента типу T тут".

**`die`** — булевий прапорець "я хочу померти". Останнім, бо смерть приходить останньою.

Зверніть увагу: `Intent` — це звичайний `dataclass` з `default_factory`, нічого магічного. Це просто **запис ефектів**. Якщо хочете інтуїцію, уявіть, що це лист до Мадам Природи: "пані Природо, я хочу рухатися на північний-схід, поставити свою енергію на 8, відкласти 0.5 феромону на клітинку, і не помирати, дякую вам".

### Дві фази: build і apply

У [`swarmlet/engine.py`](../swarmlet/engine.py) метод `_agent_phase` робить рівно дві речі:

```python
def _agent_phase(self):
    if not self.agents:
        return

    indices = list(range(len(self.agents)))
    self.rng.shuffle(indices)

    intents: List[Tuple[AgentRecord, Intent]] = []

    for idx in indices:
        agent = self.agents[idx]
        if not agent.alive:
            continue
        body = self._agent_rules.get(agent.agent_type)
        if body is None:
            continue

        intent = Intent(agent_id=agent.id)
        agent_view = agent.copy()
        ctx = self._make_agent_ctx(agent_view)

        self._eval_action(body, ctx, intent, agent_view)
        intents.append((agent, intent))

    self._apply_intents(intents)
```

**Фаза 1 (build).** Цикл по агентах у перемішаному порядку (порядок детермінований по seed-у через `self.rng.shuffle`). Для кожного агента створюється свіжий `Intent`, копія агента (`agent_view`) для ізоляції внутрішньоактовою видимості, і evaluator проходить по тілу правила, **наповнюючи інтент**. Світ не змінюється. Жодна клітинка, жодне поле, жодна координата агента — нічого. Тільки інтент пишеться. Це чиста фаза.

**Фаза 2 (apply).** `self._apply_intents(intents)` — це другий метод, який бере зібраний список і **тільки тут** мутує світ, у документованому порядку.

Розділення абсолютне. Фаза 1 ніколи не пише у світ. Фаза 2 ніколи не оцінює правила.

## Чому з цього автоматично випливає чистота, можливість резолюції конфліктів, і детермінізм

Три властивості, які кореляють у дизайнах з intent pattern, не випадково. Розглянемо кожну.

### Чистота

Правило агента — це функція `eval_action(body, ctx, intent, agent_view)`. Вона приймає AST правила, контекст з посиланням на снапшот світу, інтент (який вона наповнює) і копію агента. Вона **не має посилання, через яке могла б змінити світ**. Вона може мутувати тільки свій локальний інтент і свою локальну копію агента.

Це не просто конвенція — це структурна гарантія. Якщо хтось напише evaluator, що пробує мутувати `ctx.world`, цього просто немає у API. Метод `World.get_state(x, y)` повертає значення, але `World` не має `set_state` або іншого паблік-методу для мутації під час evaluation. Усі мутації йдуть тільки через `_apply_intents`, який викликається після build-фази.

Як наслідок: правила можна тестувати ізольовано. Можна побудувати будь-який снапшот, дати правилу пустий інтент, дати йому виконатися, і подивитися, що в інтенті. Це елементарно тестується. Без intent pattern довелося б будувати моки усього світу, інтерсептити мутації, і взагалі це ставало б грою у мокінг.

### Резолюція конфліктів

Тепер найцікавіше. Ось як виглядає `_apply_intents` (скорочено):

```python
def _apply_intents(self, intents):
    # 1. Movement conflicts
    move_targets = {}
    for agent, intent in intents:
        if intent.move_dir != STAY and not intent.die:
            dx, dy = dir_offset(intent.move_dir)
            nx, ny = self._wrap(agent.x + dx, agent.y + dy)
            if nx is not None:
                move_targets.setdefault((nx, ny), []).append((agent, intent))

    # 2. Resolve: one winner per cell
    for target, movers in move_targets.items():
        if len(movers) == 1:
            winner_agent, _ = movers[0]
            winner_agent.x, winner_agent.y = target
        else:
            winner_idx = int(self.rng.integers(0, len(movers)))
            for i, (ag, _) in enumerate(movers):
                if i == winner_idx:
                    ag.x, ag.y = target

    # 3. Agent field writes
    for agent, intent in intents:
        for fname, val in intent.agent_field_writes.items():
            agent.fields[fname] = val

    # 4. Cell field deposits (summation)
    for agent, intent in intents:
        for fname, val in intent.cell_field_deposits:
            if fname in self.fields:
                nx, ny = self._wrap(agent.x, agent.y)
                if nx is not None:
                    self.fields[fname][ny, nx] += val

    # 5. Kill intents
    # ...

    # 6. Spawn intents
    # ...

    # 7. Death intents
    for agent, intent in intents:
        if intent.die:
            agent.alive = False

    self.agents = [a for a in self.agents if a.alive]
```

Коли всі інтенти лежать перед нами у списку, **резолюція конфліктів стає тривіальною**. Хочемо щоб тільки один вовк зайняв клітинку? Будуємо `move_targets: {(x,y) -> [movers]}`, для кожного цільового місця обираємо одного переможця через `rng.integers`, інші залишаються на місці. Хочемо щоб два депозити феромону складалися? Просто `+=` всередині четвертого циклу. Хочемо щоб смерть прийшла після спавну? Робимо смерть кроком 7, спавн кроком 6.

Порядок пріоритетів зафіксований у [SPEC.md секція 2.3](../specification/Swarmlet-SPEC.md#23-tick) і у [`CLAUDE.md`](../CLAUDE.md):

> movement > field writes > deposits > kill > spawn > death

Це **політика**, окрема від механіки. Ви можете цю політику переписати, не змінюючи нічого у правилах агентів. Якщо колись вирішите, що смерть має йти перед спавном (бо мертвий батько не спавнить нащадка), просто переставите два кроки в `_apply_intents`. Жодного правила переписувати не треба, жодного агента переучувати.

### Детермінізм

Той факт, що той самий seed дає той самий снапшот, побайтово, у Swarmlet — це не побічний ефект, а наслідок такого дизайну.

По-перше, всі джерела випадковості йдуть через `self.rng` (`numpy.random.Generator` з фіксованим seed). По-друге, порядок обходу агентів у build-фазі задається `self.rng.shuffle` — детермінованим. По-третє, порядок резолюції конфліктів — фіксований і не залежить від часу обчислення (поки інтенти зібрані, вони лежать у пам'яті, не залежно від того, скільки часу зайняла їх побудова). По-четверте, build-фаза не мутує світ, тому її порядок ірелевантний для пізніших обчислень — навіть якби обчислили її паралельно (чого зараз немає, але теоретично можна), результат був би той самий, бо інтенти будуються незалежно один від одного.

Без intent pattern довелося б штучно серіалізувати агентів, заборонити паралелізм, додати mutex-и, і все одно ніщо б не гарантувало порядок. З ним — детермінізм є **природним наслідком архітектури**, а не зусиллям.

## П'ять знайомих паралелей

Якщо вам ця ідея здається знайомою — це тому, що вона і має бути. Intent pattern — це фактично конкретне втілення серії патернів, які з'являлися у різних доменах для розв'язання дуже схожих проблем. Розглянемо п'ять.

### 1. Redux actions / reducer pattern

Якщо ви писали React-додатки на Redux, intent pattern для вас — рідне дитя. У Redux є state (один великий об'єкт), є actions (плоскі об'єкти на кшталт `{type: 'INCREMENT', payload: 1}`), і є reducer-функція `(state, action) -> newState`.

Логіка компонента **не мутує** state напряму. Вона **створює action** і відправляє його в store. Store викликає reducer, reducer повертає новий state. Це гарантує, що всі мутації локалізовані в одному місці, можна логувати, можна реплеїти, можна undo/redo.

У Swarmlet `Intent` — це фактично action. `_apply_intents` — це reducer. `World` — це state. Один агент за тік генерує "набір actions". Reducer застосовує їх з пріоритетами. Та сама ідея: "обчислити, що зробити" і "застосувати" — це різні поверхні з чітким контрактом між ними.

Різниця у деталях: Redux зазвичай застосовує одну action за раз, тут — пакетом за тік. Але філософія ідентична.

### 2. Database transactions (BEGIN/COMMIT)

Транзакції у базах даних роблять те саме, тільки на рівні durability і атомарності. Ви робите `BEGIN`, потім кучу `INSERT/UPDATE/DELETE`. Усі ці зміни **накопичуються у write-ahead log** або у локальному буфері. Реальний стан таблиць не міняється. Потім ви робите `COMMIT`, і всі зміни застосовуються **атомарно** — або всі, або жодна.

Якщо посеред транзакції виникає конфлікт з іншою транзакцією (наприклад, обидві оновлюють той самий рядок), engine має політику: serializable, repeatable read, snapshot isolation тощо. Це конкретна політика резолюції конфліктів, яка визначена окремо від коду додатка.

Аналогія прозора: build-фаза = транзакція, що накопичує зміни. apply-фаза = COMMIT з резолюцією конфліктів. Snapshot isolation у базах = "правила бачать світ тільки до початку тіка".

### 3. Event sourcing

В event sourcing (наприклад у CQRS-архітектурах або в Kafka Streams) state ніколи не зберігається напряму. Замість state зберігається **журнал подій** (events): `OrderCreated`, `OrderPaid`, `OrderShipped`. Поточний state — це fold всіх подій з початку до зараз. Хочеш бачити стан вчора — фолдиш до вчорашньої події.

У Swarmlet кожний `Intent` — це по суті event. Якби ми зберігали усі інтенти всіх агентів за всі тіки в одному файлі, ми могли б повністю відновити будь-який стан світу через replay. Це не гіпотеза — це фактично як побудована система детермінізму у Swarmlet. Файл snapshot-ів JSONL (`-o out.jsonl`) — це "view" на event-stream, а сам evaluator — це projector, що проектує події у view.

Відмінність: у класичному event sourcing події зберігаються в durable log і можуть бути перепрогравані з будь-якої точки. У Swarmlet ми йдемо тік за тіком, і єдиний "log" — це послідовність snapshot-ів. Але метафора працює.

### 4. Command pattern (Gang of Four)

Command pattern — один з оригінальних патернів GoF. Ідея: інкапсулювати запит на дію в об'єкт. Об'єкт-команда має метод `execute()`, який виконує дію. Ви можете передавати команди, ставити їх у чергу, undo-їти, логувати, серіалізувати — все, що завгодно.

`Intent` — це фактично батч-команда. Він не має `execute()` (ця логіка живе в `_apply_intents`), але це питання архітектурного смаку — у функціональних мовах часто розділяють дані і їх обробку. У Java/C++ ви б, мабуть, написали `class MoveCommand`, `class SpawnCommand`, з методом `execute(world)`. У Python з dataclass-ами ми пишемо плоский запис і одну функцію застосування. Семантично це той самий патерн.

GoF мотивував Command тим, що це дозволяє відмінити дію (undo), журнал команд, makro-команди. Усе це ми отримуємо з Intent: у нас є явний список того, що мало статися, ми можемо логувати, можемо рефлектити, можемо undo-їти (якщо би хотіли).

### 5. CRDTs (Conflict-free Replicated Data Types)

Останній, найекзотичніший паралель — CRDT. Це структури даних, спроектовані для паралельних розподілених систем, де кожна нода може незалежно вносити зміни, а потім ноди синхронізуються і всі сходяться до того самого стану — **детерміновано**, незалежно від порядку синхронізації.

Як CRDT це досягає? Через **комутативні операції**. Якщо у вас лічильник, і ви хочете щоб дві ноди могли збільшувати його незалежно, ви робите його G-Counter (Grow-only Counter): кожна нода інкрементить свою локальну версію, а sum по всіх версіях — це поточне значення. Збільшення комутативне і асоціативне, тому порядок синхронізації не має значення.

Депозити у клітинні поля у Swarmlet — це фактично CRDT-style операція. `cell_field_deposits` сумуються незалежно від порядку, хто перший задепозитив. Якби порядок мав значення (наприклад, `set` замість `+=`), детермінізм був би крихкий. Через сумування — він стійкий за конструкцією.

Параллель глибша, ніж здається. Якщо ви захочете розпаралелити Swarmlet (поки не варто, але теоретично), то build-фазу можна буде виконувати на кількох потоках одночасно, бо інтенти незалежні. Apply-фаза має бути єдино-потокова, але коротка. Це структура mapreduce: map = build, reduce = apply.

## OCaml-міст: записи і функціональні апдейти

Якщо ви знайомі з OCaml або іншою ML-родиною, intent pattern має ще ближчу аналогію. У OCaml записи (records) — це фундаментальний тип, і є синтаксис для **функціонального оновлення** записів через `with`:

```ocaml
type intent = {
  agent_id : int;
  move_dir : int;
  agent_field_writes : (string * float) list;
  cell_field_deposits : (string * float) list;
  spawn_types : string list;
  kill_targets : string list;
  die : bool;
}

let empty_intent id = {
  agent_id = id;
  move_dir = -1;
  agent_field_writes = [];
  cell_field_deposits = [];
  spawn_types = [];
  kill_targets = [];
  die = false;
}

let with_move dir intent =
  { intent with move_dir = dir }

let with_field_write name value intent =
  { intent with agent_field_writes = (name, value) :: intent.agent_field_writes }

let with_die intent =
  { intent with die = true }
```

Зверніть увагу на `{ intent with move_dir = dir }`. Це створює **новий запис**, де всі поля скопійовані з оригіналу, окрім `move_dir`, який отримує нове значення. Оригінальний `intent` залишається незмінним. У ML-родині це базова мова — з неї будуються всі більш складні структури.

Тоді evaluator у функціональному стилі виглядав би приблизно так:

```ocaml
let rec eval_action action ctx intent =
  match action with
  | AStay -> intent
  | ADie -> with_die intent
  | AMove dir_expr ->
      let dir = eval_expr dir_expr ctx in
      with_move (int_of_value dir) intent
  | ASet (field_name, expr) ->
      let value = eval_expr expr ctx in
      with_field_write field_name (float_of_value value) intent
  | ASeq actions ->
      List.fold_left (fun acc a -> eval_action a ctx acc) intent actions
  | AIf (cond, then_a, else_a) ->
      if bool_of_value (eval_expr cond ctx)
      then eval_action then_a ctx intent
      else eval_action else_a ctx intent
```

`ASeq` — це особливо красиво. `List.fold_left` бере початковий інтент і прокидає його через усі дії, кожна повертає новий інтент. Жодних мутацій. Чистий fold.

У Python через mutability `Intent` ми робимо іншу варіацію: інтент мутується, і `_eval_action` повертає `None`. Семантично це той самий патерн (накопичення), але реалізаційно — стандартна для Python техніка in-place мутації локального об'єкта. У OCaml/Haskell ми б писали fold; у Python — пишемо мутацію локального запису. Результат еквівалентний.

Чому це важливо? Бо це показує, що intent pattern — не якась специфічна для Swarmlet річ, а фундаментальний шаблон функціональних мов. У OCaml він просто вписаний у стандартний стиль кодування, без зайвої декорації.

## Внутрішньоактова видимість і документований "kill quirk"

Тут є кілька тонкощів, які треба розуміти, бо вони змінюють інтуїцію.

### Видимість усередині `seq`

Подивіться на цей код з вівці:

```swl
seq {
  set energy = self.energy - 1;
  if self.energy <= 0 then die else stay
}
```

Інтуїтивно: я зменшив енергію, потім перевіряю чи вона стала ≤ 0. Але якщо `set` тільки накопичується в інтент і не міняє реального агента, то `self.energy` у другому рядку має повертати **стару** енергію, а не нову. Тобто перевірка завжди буде на старе значення.

Це не те, чого ми хочемо. SPEC говорить:

> Within-action visibility: within a single agent's action, `set` writes to the agent's own fields are visible to subsequent reads in the same `seq` block.

Як це досягається? У `_agent_phase` створюється `agent_view = agent.copy()` — локальна копія агента, окрема від реального агента у списку. Усі читання `self.energy` всередині evaluator-а беруть значення з `agent_view.fields`. Усі `set energy = ...` оновлюють і `intent.agent_field_writes`, і **`agent_view.fields[...]`**. Тобто всередині дії агент бачить свою власну енергію оновленою, а зовнішній світ і інші агенти — ні.

Це правильна семантика. Для агента це виглядає, ніби він діє "крок за кроком" локально. Для світу всі його дії — це один атомарний інтент.

**Винятково важливо:** клітинні депозити (`cell_field_deposits`) **не** мають цієї властивості. Якщо мураха зробить `set pheromone = field pheromone + 0.5`, і потім у тій же дії читає `field pheromone`, він прочитає **старе** значення, бо клітинні поля міняє тільки apply-фаза, після всіх інтентів.

### Quirk зі вбивством

З [SPEC.md секція 2.3](../specification/Swarmlet-SPEC.md#23-tick):

> If two agents both attempt to kill the same target, one succeeds and the other's `kill` is silently dropped (but its energy gain, computed earlier in its action body, is NOT undone — see section 12.4 for the documented quirk).

Згадайте сценарій з двома вовками. Обидва на одній клітинці з вівцею. Обидва обчислюють `if agents_of_type_in_radius Sheep 0 > 0 then seq { kill Sheep; set energy = self.energy + wolf_kill_gain }`. На етапі build обидва бачать `Sheep > 0` (бо світ ще не змінився), обидва додають `kill Sheep` до своїх інтентів, обидва нараховують собі `+wolf_kill_gain` енергії.

На етапі apply: один з двох `kill` випаде, бо вівця одна. Але **обидві енергії вже нараховані в інтенти** і вони обидва будуть застосовані. Тобто два вовки з'їли одну вівцю і обидва отримали повну ситість. У природі це абсурд, у симуляції — компроміс на користь простоти. SPEC документує це як known quirk у v0.1; виправити можна було б, перевіряючи success kill-у і "відмотуючи" energy gain, але це додало б складність, яка не варта виграшу для більшості моделей.

Навмисність цього вибору важлива. Це не баг, це дизайнерське рішення на користь read-only-ness build-фази. Якби ми хотіли точніше моделювати, треба було б додавати ще одну фазу між build і apply: "проміжну валідацію інтентів". Це можна зробити, але втрачається простота.

## Типові пастки

Три речі, на які варто зважати при роботі з intent pattern у Swarmlet.

**Пастка 1: думати, що `set` всередині агента негайно застосовується.** Це частково так (бачимо вище — для власних полів агента), але частково ні (для клітинних полів). Якщо ви пишете `set pheromone = field pheromone + 1.0; if field pheromone > threshold then ...`, друга частина прочитає старе значення. Якщо вам потрібно зреагувати на власне депонування, треба використовувати локальний `let`: `let new_phero = field pheromone + 1.0 in seq { set pheromone = new_phero; if new_phero > threshold then ... }`.

**Пастка 2: припускати порядок обробки агентів.** Порядок шафлиться кожен тік через seeded RNG. Не пишіть правила, що залежать від конкретного порядку. Якщо вам потрібно щоб "першими ходили хижаки" — це не Swarmlet-семантика, треба переосмислити модель.

**Пастка 3: очікувати, що агент бачить ефекти інших агентів у тому самому тіку.** Не бачить. Усі агенти у тіку працюють зі снапшотом, що став результатом cell-phase. Якщо вовк A вбив вівцю на початку тіка, вовк B все ще може "бачити" цю вівцю через `agents_of_type_in_radius`, поки не пройде apply-фаза. Це може здатися дивним, але це **правильно** — без цього не було б детермінізму незалежного від порядку.

## Вправи

### Вправа 1: трасування "two wolves chasing one sheep"

**Setup.** У вас грід 5x5, на клітинці (2,2) стоїть вівця. На (2,1) стоїть вовк A, на (2,3) стоїть вовк B. Обидва бачать вівцю через `nearest_agent_of_type_dir Sheep 2`. Вовки мають енергію 10, `wolf_kill_gain = 8`. Сід задано так, що `rng.shuffle([A, B])` дає `[A, B]`, а при tie-break у move conflict обирається indeх 0 (тобто A).

**Task.** Простежте два тіки. Опишіть для кожного:
1. Які інтенти збираються у build-фазі.
2. Що відбувається у apply-фазі по кожному з 7 кроків резолюції.
3. Стан світу після тіка: координати агентів, енергії.

**Hint.** На першому тіку обидва вовки рухаються на (2,2), але тільки один доходить (бо move conflict). На другому тіку той, хто доходить, опиняється з вівцею на одній клітинці і виконує `kill Sheep`. Інший все ще на старій позиції і знов хоче рухатися. Зверніть увагу на момент "обидва ще на сусідніх клітинках, обидва бачать вівцю" — інтенти на kill виставляються тільки якщо `agents_of_type_in_radius Sheep 0 > 0`, тобто на тій самій клітинці.

### Вправа 2: розширити `Intent` для cooldown-а

**Setup.** Ви хочете додати в систему механіку "після спавну вовк не може спавнити ще 5 тіків". Стандартний `Intent` цього не підтримує.

**Task.** Як ви додасте цей функціонал? Варіанти: (а) додати поле в `AgentRecord` `spawn_cooldown` і перевіряти його в правилі агента; (б) додати поле в `Intent` і обробляти в apply; (в) комбінація. Який з варіантів зберігає чистоту build-фази? Чому (б) гірше за (а)?

**Hint.** Чим менше функціональності кладеться в `Intent`, тим простіша apply-фаза. Усе, що можна виразити через звичайні поля агента і логіку правил, краще тримати поза інтентом. Інтент — це лише "зовнішні" ефекти на світ.

### Вправа 3: реалізувати "obstinate move"

**Setup.** Зараз при move conflict програла сторона "застрягає" на місці. Ви хочете зробити агента, який при програшу move conflict пробує зрушити випадковий напрямок (fallback move).

**Task.** Чому це **не можна** реалізувати, не змінюючи структуру `_agent_phase`? Що довелося б додати? Як це зламає чистоту build-фази?

**Hint.** Інформація "ти програв move conflict" з'являється тільки в apply-фазі. Якщо агент має на неї реагувати, треба або додати ще одну build-фазу (другу спробу), або повернути керування агенту після першого apply, або перенести fallback-логіку в apply (що змішує політику резолюції з логікою агентів). Жоден варіант не безкоштовний.

### Вправа 4: cell rules як degenerate intent pattern

**Setup.** Cell phase у Swarmlet не використовує явні `Intent`-и (подивіться на `_cell_phase` в `engine.py`). Замість цього використовуються `next_states` і `next_fields` — паралельні масиви, у які записуються нові значення.

**Task.** Чи можна вважати cell phase окремим випадком intent pattern? Який тоді "інтент" клітинки? Чому для клітинок не потрібна резолюція конфліктів, а для агентів потрібна?

**Hint.** Кожна клітинка пише тільки у свою позицію в `next_states`/`next_fields`, без перетину з іншими клітинками. Один writer на одну клітинку — резолюція не потрібна. Для агентів кілька можуть зайняти одну клітинку, кілька можуть депонувати на одну клітинку — конфлікт обов'язковий. Build/apply-розділення — те саме, але pattern спрощується до "double buffering" коли writers не конфліктують.

## Підсумок

Intent pattern у Swarmlet — це не оптимізація і не cosmetic. Це **фундаментальна архітектура**, яка робить можливим все інше: чистоту правил, повторюваність симуляцій, документовану семантику конфліктів.

Якщо коротко: правила агентів — це чисті функції, що **описують** ефекти у вигляді запису `Intent`. Окремий механізм світу збирає всі інтенти і **застосовує** їх у документованому порядку: рухи, потім записи у власні поля, потім депозити в клітинні поля, потім вбивства, потім спавни, потім смерті. Кожен крок чистий, дитерміністичний, з явним обробленням конфліктів.

Це той самий шаблон, що Redux у фронтенді, transactions у базах, event sourcing у CQRS, Command pattern у GoF, і CRDT у розподілених системах. Усі вони про одне: **відокремити опис дії від її виконання**, щоб отримати контроль над порядком, конфліктами і консистентністю.

Якщо вам колись доведеться будувати симулятор, ігровий движок, або взагалі будь-яку систему з паралельними змінами стану — згадайте intent pattern. Це маленький конструктивний крок, який віддає вам контроль над усім, що раніше здавалося хаосом.
