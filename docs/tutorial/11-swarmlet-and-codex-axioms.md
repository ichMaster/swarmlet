# Swarmlet, minilog і Codex axioms — три DSL поспіль

Це документ-синтез, не туторіал. Якщо ви читаєте його, то ви, мабуть, той самий інженер, який щойно дописав minilog, доводить до релізу Swarmlet, а на горизонті у нього вже маячать Codex axioms. Три проєкти поспіль, у трьох різних доменах, з трьома різними синтаксисами — і коли ти сидиш над четвертою ітерацією evaluator-а і відчуваєш déjà vu, варто на секунду зупинитися і запитати: а що саме повторюється?

Цей текст — спроба назвати повторювану структуру. Не для того, щоб винайти Grand Unified Theory маленьких мов (її давно винайшли, вона називається "інтерпретатори і компілятори"), а для того, щоб **перетворити мовчазну інтуїцію в явне знання**. Бо коли наступного разу хтось скаже "нам потрібен конфіг на YAML з умовами", ви вже не сядете писати ad-hoc парсер — ви впізнаєте патерн, скажете "ок, це DSL", і одразу подумаєте про лексер, AST, аналізатор і evaluator. Це і є мета.

## Опорна рамка: ви побудували три з них поспіль, що це означає?

Якщо подивитися на список ваших останніх проєктів, видно одну річ: усі три — **декларативні DSL з власним runtime**.

- **minilog** — діалект Прологу. Користувач пише факти і правила (Horn clauses), движок робить SLD resolution.
- **Swarmlet** — мова для cellular automata і agent-based моделей. Користувач пише cell rules і agent actions, движок робить tick-by-tick симуляцію.
- **Codex axioms** — декларативна система аксіом (третій проєкт у тій самій родині). Користувач описує аксіоми, движок робить висновки.

У всіх трьох випадках користувач **не пише, як обчислювати** — він пише, **що є істинним** або **що має статися за певних умов**. Решту робить движок. Це і є визначення декларативної мови.

І коли ви будуєте три такі поспіль, починає вимальовуватися спільна механіка. Усі три мають лексер. Усі три мають парсер. Усі три мають AST з frozen-dataclass-ами. Усі три мають аналізатор, який ловить помилки до запуску. Усі три мають evaluator, який обходить AST і обчислює значення (або шукає докази, або застосовує правила до сітки). Зміна на рівні поверхневого синтаксису — а на рівні архітектури те саме знову і знову.

Ця документ — про спільне ядро.

## Спільна родина: три DSL поруч

Найкоротший спосіб побачити схожість — поставити три фрагменти коду поруч і запитати: що тут спільного?

**Swarmlet (cell rule):**

```swl
let cell Tree =
  match any Fire with
  | true  -> Fire
  | false ->
      if random_below(p_ignite) then Fire
      else Tree
```

**minilog (Horn clause):**

```pl
правило предок(?x, ?y) якщо батько(?x, ?y).
правило предок(?x, ?z) якщо батько(?x, ?y) і предок(?y, ?z).
```

**Codex axiom (умовний):**

```text
аксіома consistent(?theory) якщо
  для_кожного клаузу(?theory, ?c): не суперечить(?theory, ?c).
```

Що тут спільного?

1. **Голова і тіло.** У всіх трьох є форма "лівий бік := правий бік". У Swarmlet ліворуч — патерн стану клітинки і `match`-вираз, праворуч — нове значення. У Прологу ліворуч — голова клаузу, праворуч — кон'юнкція цілей. У Codex axioms ліворуч — назва аксіоми з аргументами, праворуч — умова виконання. **Це всі форми того самого**: умова → результат.
2. **Змінні з обмеженою областю.** `?x` у Прологу, `?theory` в аксіомі, неявне `cell_state` у Swarmlet — це все локальні зв'язки, які живуть тільки у межах одного правила/клаузу/аксіоми. У Прологу за це відповідає `rename_variables`, у Swarmlet — `ExprContext.child()`, у Codex axioms — буде свій механізм скоупу.
3. **Декларативність.** Жодне з цих правил не каже movkov-у, як їх застосовувати. Користувач каже **що**, движок розв'язує **як і коли**.
4. **Pattern matching як основний механізм диспетчу.** У Swarmlet — буквально `match ... with`. У Прологу — уніфікація голови з ціллю. У Codex axioms — зіставлення аксіоми з контекстом. Це той самий механізм під різними назвами.

Якщо у вас з'явився четвертий проєкт у цій родині, ви вже знаєте, що шукати: де його "правила", де його "змінні", де його "тіло", де його "застосування".

## Спільні механізми

Тепер про те, що повторюється у реалізації.

### Механізм 1: AST + interpreter

Усі три DSL мають однакову компіляційну піпеліну: **source → tokens → AST → analysis → evaluation**. AST складається з frozen-dataclass-ів (або їхнього еквіваленту), evaluator — це функція, яка приймає AST-вузол і контекст і повертає значення.

У Swarmlet це виглядає так у [`eval.py`](../swarmlet/eval.py):

```python
def eval_expr(node: Any, ctx: ExprContext) -> Any:
    if isinstance(node, A.Num):
        return node.value
    if isinstance(node, A.Bool):
        return node.value
    if isinstance(node, A.Var):
        return _eval_var(node, ctx)
    if isinstance(node, A.BinOp):
        return _eval_binop(node, ctx)
    if isinstance(node, A.If):
        cond = eval_expr(node.cond, ctx)
        if _ensure_bool(cond, node.line):
            return eval_expr(node.then_expr, ctx)
        return eval_expr(node.else_expr, ctx)
    if isinstance(node, A.Match):
        return _eval_match(node, ctx)
    ...
```

У minilog (за документом про движок) — те саме, тільки замість `eval_expr` функція називається `solve`, і вона рекурсивно пробує знайти підстановку:

```python
def solve(goal, kb, subst, depth, max_depth):
    for fact in kb.lookup_facts(goal.functor, goal.arity):
        new_subst = unify(goal, fact.head, subst)
        if new_subst is not None:
            yield new_subst
    for rule in kb.lookup_rules(goal.functor, goal.arity):
        fresh_rule = rename_variables(rule, depth)
        head_subst = unify(goal, fresh_rule.head, subst)
        if head_subst is not None:
            yield from solve_body(fresh_rule.body, kb, head_subst, depth + 1, max_depth)
```

Дві функції зовні робить різні речі — одна обчислює вираз, друга шукає докази. Усередині — той самий патерн: розглянь тип AST-вузла, обери відповідний обробник, рекурсивно піди вглиб. Codex axioms матиме те саме ядро, просто з іншими типами вузлів.

Подивіться детальніше у [expression-evaluator-explained.md](expression-evaluator-explained.md) — там той самий патерн розписаний для Swarmlet окремо.

### Механізм 2: pattern matching як ядро диспетчу

`match` у Swarmlet, уніфікація у Прологу і application аксіом — це поверхневі форми **одного механізму**: візьми значення (або ціль), перебери набір патернів (або голів), знайди перший, що пасує, і застосуй пов'язане тіло.

Swarmlet:

```python
def _match_pattern(pattern: A.Pattern, subject: Any) -> bool:
    if pattern.kind == "wildcard":
        return True
    if pattern.kind == "ident":
        return subject == pattern.value
    if pattern.kind == "number":
        return isinstance(subject, (int, float)) and subject == pattern.value
    if pattern.kind == "bool":
        return isinstance(subject, bool) and subject == pattern.value
    return False
```

Це найпростіша форма pattern matching — без зв'язування змінних, без деструктуризації, без рекурсії. Просто "ідентифікатор vs ідентифікатор".

У Прологу той самий принцип, але потужніший — **уніфікація** вміє не тільки порівнювати, а й одночасно **знаходити підстановку**, яка робить два терми однаковими. Це той же `_match_pattern`, який раптом може повернути не тільки `True/False`, а ще й словник `{?x: значення, ?y: значення}`. Коли ви будете писати Codex axioms, у вас, найімовірніше, теж буде форма pattern matching — і питання буде в одному: "Чи будуть у моїх патернах змінні?". Якщо так — ви будуєте Prolog-like систему. Якщо ні — Swarmlet-like.

Глибше про цей перехід — у [pattern-matching-explained.md](pattern-matching-explained.md).

### Механізм 3: чисті функції і незмінні контексти

Третя спільна річ — **жоден з цих evaluator-ів нічого не мутує**. Вони приймають контекст, обчислюють результат, і повертають **новий** контекст (або значення). Ось як це виглядає у Swarmlet:

```python
def child(self, **overrides) -> ExprContext:
    new_locals = dict(self.locals)
    if "locals" in overrides:
        new_locals.update(overrides.pop("locals"))
    return ExprContext(
        rng=self.rng,
        locals=new_locals,
        ...
    )
```

`ExprContext.child()` — це функціональний `extend`. Старий контекст не змінюється; повертається новий, з доданими локальними змінними. Те саме у Прологу: `subst` ніколи не мутується. Кожен виклик `unify` повертає **новий** словник підстановок, а старий лежить незмінений на стеку — щоб коли backtracking відкотить вибір, можна було повернутися до попереднього стану без жодних "undo" операцій.

Це не випадковість і не стиль кодування "для краси". Це **технічна вимога**: щоб backtracking або branching у evaluator-і працювали правильно, контекст має бути імутабельним. Інакше доведеться вручну записувати, що змінилося, і скасовувати — і це шлях до катастрофи. Codex axioms, якщо там буде будь-яка форма пошуку (а вона, найімовірніше, буде), — теж матиме імутабельний контекст. Це не вибір, це наслідок.

## Спільні граблі

Три речі, на які ви впораєтеся знову у Codex axioms — і варто бути до них готовим.

### Грабля 1: детермінізм важко виборювати

У Swarmlet вся випадковість тече через **один seeded RNG**, який передається у `ExprContext.rng`. Якщо хоч одне місце візьме `random.random()` напряму — детермінізм зламається, і запис буде неможливо відтворити. У Прологу аналогічний біль — порядок розв'язків залежить від порядку правил у файлі, від порядку фактів, від точок вибору. Один порядок переплутали — повторити запит з тим самим результатом стає неможливо.

У Codex axioms це вилізе у формі **порядку застосування аксіом**. Якщо двічі прогнати одну й ту саму базу і отримати різні результати — інженерна катастрофа. Захист: **єдина точка входу для будь-якого недетермінізму**, і вона приймає seed.

### Грабля 2: вкладені конструкції потребують дужок

Це класична грабля у будь-якій мові з префіксними operator-ами і виразами. У Swarmlet `match a == b with | true -> ...` парситься не так, як ви думали (бо `==` має нижчий пріоритет, ніж очікується). У Прологу `не предок(X, Y), предок(Y, X)` теж може бути сюрпризом — `не` зв'язується слабше, ніж кома. У Codex axioms такий самий клас помилок виникне на стику умов і кванторів.

Захист: **парсер з явним пріоритетом операторів** і тести на помежові випадки. У Swarmlet це робиться в `parser.py` через ієрархію функцій `parse_or → parse_and → parse_compare → ...`. Той самий патерн працює і у minilog, і працюватиме у Codex axioms.

### Грабля 3: статичні vs runtime помилки

У Swarmlet є **аналізатор** — окремий прохід, який ловить недостатньо оголошені стани, неіснуючі функції, дублікати правил. Це класно, бо помилки видно одразу при `swarmlet check`, без запуску. У Прологу такого аналізатора майже немає — `undefined predicate` ловиться тільки в момент виклику, коли движок намагається знайти клаузу і не знаходить. Це особлива форма болю: вам треба прогнати симуляцію 5 хвилин, щоб дізнатися, що ви забули оголосити функтор.

У Codex axioms спокуса буде "не писати аналізатор, бо ще зарано". Не піддавайтеся. Аналізатор — це маленька машинерія за один день, але вона економить тижні дебагінгу. Свариться рано — це чесніше, ніж свариться пізно.

## Конкретний міст: forward chaining як cellular automaton

Тепер цікаве. У Прологу є **backward chaining** (типовий SLD-resolution: починаємо з цілі, шукаємо правила, що її доводять) і **forward chaining** (починаємо з фактів, застосовуємо правила, отримуємо нові факти, повторюємо до фіксованої точки). Перший — це класичний Пролог. Другий — це **експертні системи типу OPS5 і Drools**.

А тепер придивіться, що робить Swarmlet на кожному tick-у. Він бере поточний стан світу, **застосовує всі правила паралельно**, отримує новий стан, повторює. Якщо правила побудовані так, що рано чи пізно стан перестане змінюватися (наприклад, "ліс згорів повністю, нових пожеж немає") — Swarmlet природно прийде до **фіксованої точки**.

Це і є forward chaining. Дослівно. Cellular automaton — це **forward-chaining engine з 2D-структурою фактів**. Кожна клітинка — це факт `cell(x, y, state)`. Кожне cell rule — це Horn clause виду `cell(x, y, NewState) :- cell(x, y, OldState), neighbors(...)`. Кожен tick — це один прохід forward chaining по всім фактам.

Чому це корисно знати? Бо коли ви будете будувати Codex axioms з forward chaining (а аксіоми майже завжди працюють forward), ви зрозумієте, що **архітектурно це той самий движок, що Swarmlet**, тільки з абстрактнішими "клітинами" (предмет аксіоми замість координат). Можна буквально перевикористати інтуїцію tick-а: phase 1 — обчислити всі застосовні аксіоми; phase 2 — застосувати атомарно. Це той самий два-фазний детермінований tick, що описаний у [tick-as-snapshot-transformation.md](tick-as-snapshot-transformation.md).

## Конкретний міст: cell states як Horn clause atoms

Друга паралель, яка стає очевидною, коли її назвати, але не очевидна заздалегідь.

Стан клітинки у Swarmlet — це атом. `Tree`, `Fire`, `Empty` — це не структури, не записи, не об'єкти. Це **символи**. Точно так само, як у Прологу `ісак`, `авраам`, `яків` — це атоми. Жодних полів, жодних методів, тільки ідентичність.

І коли ви робите `state(x, y)` у Swarmlet, ви питаєте "якого кольору клітинка". Це еквівалентно Prolog-запиту `?- cell_state(X, Y, ?s).` — "який атом приписаний до пари координат". Якщо подивитися на Swarmlet через призму Прологу, він — це **спеціалізована Prolog-база, де всі факти мають форму `cell_state(x, y, atom)`**, а движок надзвичайно ефективно зберігає її у numpy-масиві замість загального indexing-у по `(functor, arity)`.

Те саме працює у зворотний бік: якщо вам треба порахувати, скільки сусідніх клітинок мають стан `Tree`, у Прологу ви напишете:

```pl
правило count_trees(?x, ?y, ?n) якщо
  findall(?nx-?ny, (сусід(?x, ?y, ?nx, ?ny), cell_state(?nx, ?ny, tree)), ?list),
  довжина(?list, ?n).
```

У Swarmlet — просто `count(Tree)`. Але семантично це те саме обчислення. Codex axioms майже напевно матиме якусь форму "перерахуй усі об'єкти, що задовольняють умову" — і коли ви будете її проєктувати, корисно тримати в голові, що це той самий `count` з нестандартним фільтром.

Цей зв'язок добре підкреслює одну важливу річ: **алгебраїчні типи у Swarmlet** ([algebraic-data-types.md](algebraic-data-types.md)) — це не що інше, як скінченна множина атомів, до яких можна застосовувати pattern matching. У Прологу ці атоми не оголошуються — вони просто існують. У Swarmlet їх треба оголосити через `state Tree | Fire | Empty;`. У Codex axioms вибір залежатиме від того, наскільки строго ви хочете типізувати — обидва варіанти мають свої плюси.

## Мета-вміння: розпізнавати "це насправді маленька мова"

Тепер про найголовніше. Все, що описано вище — це конкретні механізми. Але є одне **мета-вміння**, яке три проєкти поспіль виховують у голові, і яке варто назвати явно.

**Коли ви робите конфіг-систему з умовами, rule engine, шаблонізатор з логікою, expression evaluator у конфізі — ви будуєте DSL, незалежно від того, визнаєте ви це чи ні.**

Це звучить тривіально, але насправді це **найдорожчий урок** у роботі з декларативними системами. Більшість проєктів починаються з фрази "нам не треба парсер, просто кілька умов у YAML". Через шість місяців у YAML живуть умови з `${...}` змінними, з `if/then/else`, з функціями типу `concat()` і `map()`. Через рік це повноцінна Тьюрінг-повна мова **без специфікації, без аналізатора, без тестів і без документації**. Інженери, які це будували, ніколи не визнавали, що пишуть мову — вони "просто додавали маленьку фічу".

Сигнали, що ви насправді будуєте DSL, а не "просто конфіг":

- Користувач має зв'язувати імена зі значеннями (`let`, `var`, `${name}`).
- Користувач має писати умови (`if`, `when`, `match`).
- Користувач має комбінувати значення (`+`, `&&`, `concat`).
- Користувач має посилатися на інші частини конфігу (рекурсія, посилання).
- Користувачам не вистачає вашого синтаксису і вони просять "трохи розширити".

Якщо ви бачите хоча б три з цих сигналів — **зупиніться і визнайте, що це DSL**. Дайте йому ім'я. Напишіть лексер. Напишіть парсер. Напишіть AST. Напишіть аналізатор. Напишіть evaluator. Це **тиждень роботи**, і ви отримуєте систему, яку можна підтримувати. Якщо не зробите — ви отримуєте 3000 рядків коду на регулярних виразах, який ніхто не наважується чіпати.

Три проєкти поспіль — це достатньо, щоб у голові намалювалося **готове ядро**: "ну я знаю, як це робити". Скористайтеся з цього. У Codex axioms ця інтуїція спрацює одразу, на перших днях, і збереже тижні роботи. Це і є справжня цінність повторення.

## Підсумок

Три проєкти поспіль виявили, що ви робите варіації одного й того самого: **декларативний DSL з власним runtime**. Архітектура — лексер, парсер, AST, аналізатор, evaluator — повторюється з мінімальними варіаціями. Pattern matching — і поверхневий `match`, і уніфікація — це той самий механізм диспетчу. Імутабельні контексти — не стиль, а технічна необхідність для backtracking-у або багатофазних tick-ів.

Спільні граблі — детермінізм, дужки навколо вкладень, ранній vs пізній звіт про помилки — теж повторюються, і у Codex axioms вони вилізуть знову. Знаючи це наперед, можна закрити їх за тиждень, а не за квартал.

І найголовніше — **мета-вміння**: розпізнавати, що "конфіг", "rule engine", "шаблон з логікою" — це маленькі мови. Коли ви це впізнаєте, ви автоматично переходите з режиму "хакай швидко" у режим "проєктуй мову". Це і є те, що повторення трьох DSL дає назавжди.

Якщо коротко: ви перестали дивуватися, коли треба написати парсер. Це вже навичка, не подія. У Codex axioms вона спрацює сама, без зусиль — а звільнений час можна витратити на те, що справді унікально для аксіом, а не на повторне винайдення колеса.
