# `let`-вирази проти змінних: чому `let x = 5 in expr` — це НЕ присвоєння

> Перед цим документом варто прочитати [fp-mental-model.md](fp-mental-model.md), щоб мати загальну картину функціонального мислення. Споріднений Tier 1 документ — [pattern-matching-explained.md](pattern-matching-explained.md), який пояснює `match ... with`. Формальна семантика `let` в Swarmlet — це [SPEC.md секція 7.5](../specification/Swarmlet-SPEC.md#75-let-expressions).

Цей документ — про одну маленьку синтаксичну конструкцію, яка вводить в оману майже всіх, хто приходить у функціональне програмування з Python, Java або SQL. Конструкція виглядає як знайома стара подруга — присвоєння змінній. Але вона нею не є. І коли ви це відчуєте на кінчиках пальців, у вас несподівано зміняться інтуїції про багато речей: про рефакторинг, про композицію, про те, як читати чужий код.

## Зав'язка: щоденне непорозуміння

Ось ситуація, яка виникає у кожного, хто перший раз бачить рядок коду на Swarmlet.

```swl
let cell Empty =
  let grow = random () < growth_rate in
  if grow then Tree else Empty
```

Ви читаєте це і думаєте: "ага, тут створюється змінна `grow`, в неї записується результат порівняння, а потім ми її використовуємо в `if`". Це зрозуміло. Це ж як у Python:

```python
def cell_rule_empty():
    grow = random() < growth_rate
    if grow:
        return Tree
    return Empty
```

І начебто все логічно — є оператор `let`, який створює локальну змінну, і потім ця змінна доступна нижче. Майже як `=` у Python, тільки з ключовим словом. Окей, рухаємось далі.

А потім ви натикаєтесь на щось таке:

```swl
let cell _ =
  let cu  = field u in
  let cv  = field v in
  let lu  = laplacian u in
  let lv  = laplacian v in
  let uvv = cu * cv * cv in
  let du  = Du * lu - uvv + feed * (1.0 - cu) in
  let dv  = Dv * lv + uvv - (feed + kill) * cv in
  seq {
    set u = clamp (cu + du) 0.0 1.0;
    set v = clamp (cv + dv) 0.0 1.0
  }
```

Сім `let` підряд, і всі вони з тим самим словом `in`. Чому `in`? Що означає `let cu = field u in let cv = field v in ...`? У Python ви б просто написали сім окремих рядків, без жодного `in`. Звідки воно тут і навіщо?

Ось тут і починається непорозуміння. Бо `let ... in ...` — це не сім окремих присвоєнь. Це **один великий вираз**, всередині якого ви назвали сім підвиразів.

І це не педантизм. Це інша модель мислення. Та сама, що відрізняє SQL-запит від послідовності INSERT-ів. Та сама, що відрізняє математичну формулу від алгоритму. І коли ви її відчуєте, конструкція `let ... in ...` стане вашим найкращим інструментом для написання чистого коду.

## Серцевина: вираз проти інструкції

Щоб зрозуміти `let ... in ...`, треба спочатку зрозуміти різницю між **інструкцією** (statement) і **виразом** (expression). Це фундаментальне розрізнення, яке у багатьох імперативних мовах розмите, а у функціональних — гостре і явне.

### Інструкція — це команда машині щось зробити

Інструкція щось **робить**. Вона змінює стан світу. У Python:

```python
x = 5         # інструкція: записати 5 в комірку пам'яті, названу x
print(x)      # інструкція: вивести x на екран
x = x + 1     # інструкція: прочитати x, додати 1, записати назад
```

Коли ви виконуєте інструкцію, у пам'яті щось змінюється. Після `x = 5` десь у RAM лежить число 5, а ім'я `x` тепер вказує на нього. Якщо я запитаю "який результат `x = 5`?", у Python немає осмисленої відповіді. Інструкція не має значення. Вона має **ефект**.

### Вираз — це опис, який має значення

Вираз — це щось, що можна **обчислити** і отримати значення. У Python:

```python
2 + 3                    # вираз, обчислюється у 5
random() < 0.5           # вираз, обчислюється у True або False
"hello" + " " + "world"  # вираз, обчислюється у "hello world"
```

Кожен вираз має значення. Якщо я запитаю "який результат `2 + 3`?", відповідь — `5`. Це тип. Це число.

Тепер фокус: **у Swarmlet усе є виразом**. Немає інструкцій взагалі. Програма на Swarmlet — це купа правил, кожне з яких є виразом, що обчислюється у нову клітинку або у дію агента. У ній не можна "виконати" `x = 5` як окрему дію, бо ця конструкція просто не існує.

### І тут з'являється `let ... in ...`

Якщо все є виразом, то як назвати підвираз? У Python ви б просто написали:

```python
y = some_expensive_computation(x)
z = y * y + 1
```

Дві інструкції. Перша записала результат у `y`, друга використала `y`. Але у Swarmlet немає інструкцій. Тоді як це зробити?

Ось як: вводимо новий вид виразу — `let`-вираз. Він має таку форму:

```
let <ім'я> = <вираз1> in <вираз2>
```

Що це робить? Воно обчислює `<вираз1>`, дає результату ім'я, і потім обчислює `<вираз2>` — у якому це ім'я тепер можна використовувати. **Значенням всього `let ... in ...` є значення `<вираз2>`**.

Тобто `let y = 5 in y + 1` — це вираз, значення якого `6`. Не "інструкція, що створює змінну `y`, а потім окремо обчислює `y + 1`". А **один вираз із значенням 6**, у якого є локальна назва `y` для проміжного обчислення.

### Коротко, як формула

Подумайте про математичну нотацію:

> Нехай `y = 5`. Тоді `y + 1`.

Це не "запиши 5 у блокнот, потім додай 1". Це опис: "значення виразу `y + 1` для випадку, коли `y` дорівнює 5". Математик читає це як одне ціле — "дорівнює 6". Він не уявляє жодної комірки пам'яті, у яку щось записують.

Слово "let" у математиці означає "let's say" — "припустімо, нехай". Це введення позначення для зручності читання, а не модифікація світу. Так само і у Swarmlet (і в OCaml, і в Haskell, і в SML, і в інших ML-сімейних мовах). `let x = 5 in expr` означає "у виразі `expr` хай `x` означає `5`. Чому дорівнює `expr`? Ось чому".

### Альтернатива: іменування підвиразу прямо в самому виразі

Подумайте про `let x = 5 in expr` як про синтаксичний цукор для такого:

```
expr (з усіма входженнями x, заміненими на 5)
```

Тобто `let x = 2 + 3 in x * x` — це насправді спосіб написати `(2 + 3) * (2 + 3)`, але без повторення обчислення двічі. Ви даєте підвиразу `2 + 3` ім'я `x`, щоб не писати його двічі і щоб читачу було легше зрозуміти, що це за число.

`let` — це механізм **назв для підвиразів**. Не "змінних". Підвиразів. Якщо у вас у формулі тричі зустрічається `field u`, ви можете дати цьому підвиразу ім'я `cu` і використати тричі коротке `cu`. Це не пам'ять. Це ергономіка читача.

## Як це виглядає у Swarmlet

Тепер, коли модель прояснилась, подивимось на конкретні особливості: lexical scoping, immutability і shadowing. Усі три — наслідки того, що `let` — це вираз, а не інструкція.

### Lexical scoping (лексична область видимості)

**Лексична область видимості** означає: де ім'я доступне, видно з тексту програми. Не з того, як виконання дійшло до цієї точки. Не з якогось динамічного контексту. А з того, де у тексті стоїть `let` і де його `in`.

```swl
let cell Empty =
  let grow = random () < growth_rate in
  if grow then Tree else Empty
```

Тут `grow` доступне тільки всередині того, що йде після `in` — тобто всередині `if grow then Tree else Empty`. Поза цим — імені `grow` не існує. Якщо ви напишете десь нижче в іншому правилі `if grow then ...`, аналізатор скаже "невідоме ім'я `grow`". Бо текстуально воно не у тій області.

Це дає чудову властивість: щоб зрозуміти, що означає `grow`, не треба бігати по всьому файлу. Достатньо подивитись угору від місця використання — і знайти найближчий охоплюючий `let grow = ... in`. Все. Більше ніде ім'я не визначається. Це називається "локальне міркування" (local reasoning), і це одна з причин, чому функціональний код часто легше читати, ніж імперативний з глобальними змінними.

Ось приклад вкладеності, де область видимості важлива:

```swl
let cell Tree =
  let spark = random () < ignition_rate in
  if any Fire then Fire
  else
    (let extra = spark and (count Tree > 4) in
     if extra then Fire else Tree)
```

Тут є два `let`. Перший — `spark` — доступний у всьому "тілі" зовнішнього `let`. Другий — `extra` — доступний тільки всередині дужок, у виразі `if extra then Fire else Tree`. Поза цими дужками `extra` не існує. Якщо ви спробуєте використати `extra` десь у `else` після дужок, отримаєте помилку.

Лексична область видимості — це властивість тексту, а не виконання. Якщо ви скопіюєте фрагмент `let spark = ... in <тіло>` в інше місце, він поведеться там точно так само, як тут — за умови, що всі вільні імена у `<тілі>` визначені у новому місці. Це дуже потужно для рефакторингу.

### Immutability (незмінність)

**`let`-binding не можна "змінити"**. Бо там немає чого міняти — це не комірка пам'яті, це назва для підвиразу.

У Python це нормально:

```python
x = 5
x = 10        # окей, тепер x — це 10
x = x + 1     # окей, тепер 11
```

Кожен рядок — це окрема інструкція, що перезаписує комірку.

У Swarmlet **такого зробити не можна**. Якщо ви напишете:

```swl
let x = 5 in
let x = 10 in
...
```

— то це не "записати 5 в `x`, потім перезаписати на 10". Це створення двох різних `let`-виразів, де другий **затіняє** перший (це shadowing, про нього зараз буде). Перший `x` нікуди не "діється" — він просто стає недосяжним з того місця і нижче, бо нове ім'я `x` його перекриває.

Чому це добре? Бо коли ви бачите `let x = 5 in <тіло>`, ви знаєте, що **усередині `<тіло>` `x` буде дорівнювати 5 і нічому іншому**. Воно не зміниться у середині. Не з'явиться функція, яка "оновить" його. Не буде хитрого ефекту, коли цикл прокрутився і `x` стало іншим. `x = 5`, крапка. Ви можете тримати це у голові як абсолютну істину, поки читаєте `<тіло>`.

У контексті паралельного або декларативного коду це особливо цінно. Якщо `x` ніколи не міняється, два потоки можуть читати його одночасно без жодних блокувань. Якщо `x` ніколи не міняється, оптимізатор може підставити його значення прямо в місце використання. Якщо `x` ніколи не міняється, ви можете міркувати про код як про математику.

### Shadowing (затінення)

**Shadowing** — це коли вкладений `let` оголошує ім'я, яке вже визначене у зовнішньому `let`. Внутрішнє ім'я "затіняє" зовнішнє: усередині внутрішнього `let` ім'я означає внутрішнє значення, а не зовнішнє.

```swl
let cell _ =
  let x = 1 in
  let body =
    (let x = 2 in x + 10)   # тут x = 2 (затінено)
  in
  body + x                  # а тут x = 1 (зовнішнє повернулось)
```

Зовнішнє `x = 1`. Усередині дужок ми оголошуємо нове `x = 2`. У виразі `x + 10` після другого `let` ім'я `x` означає `2`, тому це 12. Але **поза тими дужками** друге `x` зникає, і `x` знову означає `1`. Тому останнє `body + x` — це `12 + 1`, тобто `13`.

Це не модифікація. Це не "перезапис". Це **перекриття у певній області видимості**, яке зникає на виході з цієї області. Це дуже схоже на те, як параметр функції затінює змінну з зовнішнього scope у Python:

```python
x = 1
def f():
    x = 2       # це не зовнішнє x, це нове локальне x
    return x
print(f())      # 2
print(x)        # все ще 1
```

У Swarmlet shadowing працює саме так — як аналог локального параметра, тільки введений через `let`.

**Чому shadowing корисний?** Дві поширені ситуації.

Перша — коли у формулі є тимчасові обчислення, і вам вигідно мати "версії" одного й того ж концептуального значення. Наприклад: `let x = compute1 in let x = clamp x 0 100 in use x`. Тут "x після кліпування" замінює "x до кліпування", і нижче ми працюємо тільки з кліпованою версією. Не плутаємось між двома схожими іменами.

Друга — у нащадних областях, коли ви не хочете випадково зачепити зовнішню змінну. У Swarmlet **заборонено** затіняти built-in імена (це статична помилка), але затіняти власні `let`-binding-и можна вільно.

## Walkthrough: `let`-ланцюжок у Gray-Scott

Тепер повернімось до того лякаючого коду з семи `let` підряд і розберімо його повільно. Ось він іще раз, з [`gray_scott.swl`](../swarmlet/examples/gray_scott.swl) (див. також [SPEC.md секція 11.5](../specification/Swarmlet-SPEC.md#115-gray-scott)):

```swl
let cell _ =
  let cu  = field u in
  let cv  = field v in
  let lu  = laplacian u in
  let lv  = laplacian v in
  let uvv = cu * cv * cv in
  let du  = Du * lu - uvv + feed * (1.0 - cu) in
  let dv  = Dv * lv + uvv - (feed + kill) * cv in
  seq {
    set u = clamp (cu + du) 0.0 1.0;
    set v = clamp (cv + dv) 0.0 1.0
  }
```

Прочитаймо це як математичну формулу, а не як програму.

`let cell _ = ...` — це правило для будь-якої клітинки (`_` означає "будь-який стан"). Тіло правила — один великий вираз. Цей вираз — це сім вкладених `let`-виразів, які закінчуються блоком `seq`, що описує дві дії: оновити поле `u` і оновити поле `v`.

Що робить кожен крок?

- `let cu = field u in ...` — даємо ім'я `cu` поточному значенню поля `u` у цій клітинці. Це число.
- `let cv = field v in ...` — те саме для поля `v`. `cv` — число.
- `let lu = laplacian u in ...` — лапласіан поля `u` в цій клітинці. Це обчислюється через сусідів і теж дає число.
- `let lv = laplacian v in ...` — лапласіан поля `v`.
- `let uvv = cu * cv * cv in ...` — це проміжний продукт у формулі реакції. Він з'являється у двох місцях нижче (`du` і `dv` обидва його використовують), тому ми даємо йому ім'я, щоб не обчислювати двічі і щоб формула була чистіша.
- `let du = Du * lu - uvv + feed * (1.0 - cu) in ...` — приріст для `u` за один крок Гарі-Скотта.
- `let dv = Dv * lv + uvv - (feed + kill) * cv in ...` — приріст для `v`.

І нарешті:

```
seq {
  set u = clamp (cu + du) 0.0 1.0;
  set v = clamp (cv + dv) 0.0 1.0
}
```

— нове значення `u` дорівнює старому `cu` плюс приріст `du`, обмежене у [0, 1]. Аналогічно для `v`.

**Ключове питання: чому `let`-и, а не одна гігантська формула?**

Подивіться, як виглядав би `set u`, якби ми все підставили без `let`:

```swl
set u = clamp ((field u) + (Du * (laplacian u) - (field u) * (field v) * (field v) + feed * (1.0 - (field u)))) 0.0 1.0
```

Жах. По-перше, `field u` обчислюється тричі (а в реальній формулі — більше разів). По-друге, нічого не зрозуміло. По-третє, помилку у ній майже неможливо знайти оком.

`let`-ланцюжок робить три речі одночасно:

1. **Дає імена концептам.** `cu` — це "current u". `lu` — "laplacian of u". `uvv` — "u-v-v term". Кожне ім'я — це маленький коментар, який описує, що тут обчислюється. Читач, який знає рівняння Гарі-Скотта, миттєво впізнає `uvv`.
2. **Уникає повторних обчислень.** `field u` обчислюється рівно один раз — там, де ми його іменуємо як `cu`. Потім ми використовуємо `cu` скільки завгодно разів, без повторного звернення до поля.
3. **Лінеарізує думку.** Ви читаєте зверху вниз: спочатку прочитуємо стан, потім обчислюємо проміжні величини, потім обчислюємо приріст, потім записуємо результат. Це майже як кулінарний рецепт — крок за кроком, без зворотних посилань.

Зверніть увагу: попри сім `let`-ів, **усе це — один вираз**. Якщо ви запитаєте "чому дорівнює `let cell _ = ...`?", відповідь — це блок `seq { ... }`, тобто послідовність дій. `let`-и не є кроками виконання — вони є **назвами для частин одного великого виразу**.

## OCaml-bridge: майже літерально те саме

Swarmlet запозичує синтаксис з ML-сімейних мов, особливо OCaml. Конструкція `let ... in ...` у OCaml виглядає практично так само:

```ocaml
let cell_rule () =
  let cu  = field "u" in
  let cv  = field "v" in
  let lu  = laplacian "u" in
  let lv  = laplacian "v" in
  let uvv = cu *. cv *. cv in
  let du  = du_coef *. lu -. uvv +. feed *. (1.0 -. cu) in
  let dv  = dv_coef *. lv +. uvv -. (feed +. kill) *. cv in
  set_u (clamp (cu +. du) 0.0 1.0);
  set_v (clamp (cv +. dv) 0.0 1.0)
```

(У OCaml оператори для float мають крапку: `*.`, `+.`, `-.` — це деталь, яка нас зараз не цікавить.)

Це той самий принцип. `let` — це не присвоєння. У OCaml `let x = 5 in expr` теж є виразом зі значенням. У OCaml також не можна змінити `let`-binding після його створення (мутабельний стан — це окремий механізм через `ref`). Shadowing у OCaml працює так само:

```ocaml
let x = 1 in
let x = x + 1 in     (* це нове x, що дорівнює старому + 1 = 2 *)
let x = x * 10 in    (* іще одне нове x = 20 *)
x                    (* результат: 20 *)
```

Тут чотири рази зустрічається `x`, але це **чотири різні x**, які затіняють одне одного. Перше — це `1`. Друге — це нове ім'я зі значенням `1 + 1 = 2`. Третє — нове ім'я зі значенням `2 * 10 = 20`. Четверте — використання останнього. Жодного "перезапису комірки" не відбувається.

У OCaml також є форма `let x = 5;;` без `in` — це **toplevel**-binding на верхньому рівні модуля. Він поводиться як декларація на рівні файлу, а не як вираз. Але всередині функцій ви завжди використовуєте `let ... in ...` — як вираз. Swarmlet взяв саме цю, виразну форму.

**Чому варто дивитись на OCaml для розуміння Swarmlet?** Бо OCaml — це зріла мова з великою спільнотою, і ідіоми написання чистого функціонального коду там добре відпрацьовані. Якщо ви бачите якесь незрозуміле місце в Swarmlet, корисно подумати "а як би це виглядало в OCaml?" і подивитись приклади на OCaml. Як правило, відповідь буде та сама.

Є ще одна паралель, яку варто зрозуміти. У Haskell є аналогічна конструкція `let ... in ...`, але є й варіація `where`:

```haskell
cellRule = result
  where
    cu  = field "u"
    cv  = field "v"
    lu  = laplacian "u"
    result = clamp (cu + du) 0.0 1.0
    du  = lu - cv*cv
```

Це інший синтаксичний цукор для тієї самої ідеї — назвати підвирази, але оголосити їх **після** використання. В Swarmlet такого немає, але корисно знати, що сама ідея "назв для підвиразів" має багато облич у різних функціональних мовах.

## Python parallel: локальні змінні без мутації — це і є `let`

Тепер, коли ви відчули OCaml-сторону, повернімось до Python — але з новим поглядом. Подивіться на цей код:

```python
def cell_rule_gray_scott():
    cu  = field('u')
    cv  = field('v')
    lu  = laplacian('u')
    lv  = laplacian('v')
    uvv = cu * cv * cv
    du  = Du * lu - uvv + feed * (1.0 - cu)
    dv  = Dv * lv + uvv - (feed + kill) * cv
    new_u = clamp(cu + du, 0.0, 1.0)
    new_v = clamp(cv + dv, 0.0, 1.0)
    return new_u, new_v
```

Якщо у цьому коді **жодна змінна ніколи не перезаписується**, то він семантично еквівалентний `let`-ланцюжку зі Swarmlet. Кожен `=` тут можна прочитати як "хай `cu` означає `field('u')`". Послідовність — це порядок назв, а не порядок ефектів.

І це насправді ключова інтуїція: **локальна змінна Python, до якої ви присвоюєте рівно один раз, поводиться як `let`-binding**. Жодної семантичної різниці немає. Різниця починається тоді, коли ви робите `cu = cu + 1` десь нижче — у Swarmlet такого синтаксично написати не можна, а в Python ви можете і часто робите.

Тому: якщо ви хочете писати "функціональний Python", достатньо одного правила — **присвоюйте локальній змінній рівно один раз і ніколи не перезаписуйте**. Ваш код стане структурно ізоморфним до Swarmlet (і до OCaml, і до Haskell).

Багато досвідчених Python-розробників прийшли до цього стилю емпірично — через те, що "одноразові" змінні легше тримати у голові і легше тестувати. Виявляється, вони перевідкрили `let`-вираз. Тільки без явного синтаксису.

Цікавий побічний наслідок: list comprehension у Python — це теж форма `let`. `[x*x for x in range(10)]` локально іменує `x` для кожної ітерації, і це ім'я не "залишається" після comprehension. Це той самий механізм лексичної області видимості і відсутності мутації.

## Чому це важливо на практиці

Гаразд, концептуально розібрались. Але навіщо все це у щоденній роботі? Що вам дає те, що `let` — це вираз?

### Equational reasoning (рівняннєва логіка)

Якщо `let x = e1 in e2` означає просто "значення `e2`, де `x` дорівнює значенню `e1`", то ви можете **підставляти** `e1` замість `x` без зміни значення цілого. Це називається referential transparency — посилальна прозорість.

У Swarmlet:

```swl
let x = 2 + 3 in x * x
```

еквівалентно:

```swl
(2 + 3) * (2 + 3)
```

еквівалентно:

```swl
5 * 5
```

еквівалентно:

```swl
25
```

Кожен крок — заміна рівного на рівне. Як у математиці. Ви можете робити такі заміни механічно, не задумуючись, чи не зміниться семантика. У імперативному коді так робити **не можна**, бо `x` могло мати побічні ефекти або бути модифікованим між використаннями.

Equational reasoning — це не академічна забавка. Це **спосіб налагоджувати код**: ви беруте складний вираз і поступово замінюєте підвирази на простіші, поки не побачите помилку. Або поки не побачите, що все правильно.

### Composability (композовність)

Виразу можна дати ім'я, перенести його в іншу функцію, перенести в інший файл — і він поводитиметься так само. Інструкція має сенс тільки у контексті потоку виконання. Вираз — у будь-якому контексті, де визначені його вільні імена.

Це означає, що рефакторинг "винеси цей шматок у named binding" завжди безпечний. У Python, якщо ви виносите шматок коду у функцію, треба думати, які змінні читаються, які пишуться, чи є побічні ефекти. У Swarmlet — просто загорніть у `let name = ... in ...` або у нову функцію, і все.

### Refactoring safety

Уявіть, що ви робите цей рефакторинг у Python:

```python
# до
result = compute(x) + compute(x)

# після
y = compute(x)
result = y + y
```

Безпечно? Залежить від того, чи `compute` має побічні ефекти. Якщо `compute(x)` друкує щось на екран, дві версії поводяться по-різному (друкує двічі або один раз). Якщо `compute(x)` модифікує глобальний лічильник, лічильник збільшиться на різну кількість.

У Swarmlet **усе чисте**. Тому такий рефакторинг безпечний завжди. Ви можете виносити підвирази в `let`-binding-и, об'єднувати, переставляти — поки збережена структура залежностей, поведінка не зміниться. Це дає вам сміливість рефакторити, бо ризик зламати щось мінімальний.

## Поширені помилки

Є три речі, на яких новачки спотикаються з `let ... in ...`. Розглянемо кожну.

### Пастка 1: забути `in`

У Swarmlet `let x = e1 in e2` — це **повна** форма. Не можна писати:

```swl
let cell _ =
  let cu = field u
  let cv = field v
  cu + cv
```

— парсер скаже, що чекав `in` після `field u`. Помилка зрозуміла, але часта, бо рука звикла з Python писати окремі рядки без жодних "сполучних" слів. У Swarmlet `in` — обов'язкове, бо без нього не зрозуміло, де закінчується значення і де починається тіло.

Правильно:

```swl
let cell _ =
  let cu = field u in
  let cv = field v in
  cu + cv
```

(Зверніть увагу: переноси рядків після `in` — це просто естетика, вони не змінюють парсинг. `let cu = field u in let cv = field v in cu + cv` — синтаксично еквівалентно.)

### Пастка 2: думати, що `let` створює змінну, яку можна перезаписати

```swl
let cell _ =
  let x = 5 in
  let x = x + 1 in     # це НЕ "x тепер 6", це нове ім'я x, що затіняє старе
  ...
```

Друге `let x = x + 1 in ...` — це не "x збільшилось на 1". Це **нове** binding на ім'я `x`, значення якого обчислене з посиланням на **старе** `x`. Семантично рівнозначно `let x_new = x_old + 1 in ...`.

Чому це не має значення на практиці? Бо результат той самий: нижче під цим `let` ім'я `x` означає 6. Але **концептуально** важливо розуміти, що ви не модифікували комірку — ви створили нову прив'язку. У більшості випадків різниці у поведінці немає, але якщо ви пишете рекурсивну функцію або працюєте з замиканнями (closures), різниця виявиться. У Swarmlet v0.1 замикань як таких немає, але звичка мислити "це нове ім'я, а не модифікація" — корисна.

### Пастка 3: спроба shadowing built-in імен

Ось специфічна для Swarmlet помилка. У SPEC.md прямо сказано: "It is a static error to shadow a built-in name". Тобто:

```swl
let cell _ =
  let random = 0.5 in    # ПОМИЛКА: random — це built-in
  if random < 0.1 then ... else ...
```

Аналізатор це впіймає і скаже "shadowing built-in name `random`". Чому це заборонено? Бо людина, читаючи нижче `random`, очікує, що це функція, яка повертає випадкове число. Якщо ви її затіните локальним числом, читач буде заплутаний, і код стане крихким.

Затіняти власні `let`-і — можна. Затіняти built-in-и — не можна. Це одне з небагатьох правил, де Swarmlet робить вибір на користь читача коду, а не максимальної гнучкості автора.

## Вправи

Чотири задачі для відпрацювання. Робіть на папері або у файлі — просто щоб відчути.

### Вправа 1: переписати на `let`

**Setup:** ось формула для одного кроку оновлення поля у клітинковому автоматі:

```swl
let cell _ =
  set temp = (field temp) + 0.1 * ((laplacian temp) + 0.05 * (1.0 - (field temp)))
```

**Завдання:** перепишіть цю формулу через `let`-ланцюжок, давши осмислені імена кожному підвиразу. Цільовий результат — `set temp = ...` стає короткою лінією, яка читається як кулінарний рецепт.

**Підказка:** `field temp` зустрічається двічі. Дайте йому ім'я, наприклад `t`. `laplacian temp` — це "дифузія", назвіть `lap`. `0.1 * (...)` — це "приріст", назвіть `delta`.

### Вправа 2: знайти scope

**Setup:** ось вираз із вкладеними `let`:

```swl
let cell _ =
  let a = 1 in
  let b =
    (let a = 10 in
     let c = a + 5 in
     c * 2)
  in
  a + b
```

**Завдання:** обчисліть значення цього виразу вручну. Запишіть, чому на кожному кроці `a` означає те, що означає.

**Підказка:** перший `a = 1`. Усередині дужок другий `let a = 10` затіняє його — там і `c = a + 5` означає `c = 10 + 5 = 15`. Після дужок другий `a` зникає, повертається перший.

### Вправа 3: рефакторинг повторюваного підвиразу

**Setup:** ось правило, де `count Tree` обчислюється тричі:

```swl
let cell Empty =
  if count Tree > 5 then
    (if random () < 0.1 then Tree else Empty)
  else
    if count Tree > 2 then
      (if random () < 0.05 then Tree else Empty)
    else
      if count Tree > 0 then Empty
      else Empty
```

**Завдання:** виділіть `count Tree` у `let`-binding на самому верху правила і перепишіть так, щоб обчислення сусідів відбувалось рівно один раз. Логіку зберегти.

**Підказка:** `let neighbours = count Tree in <тіло>`, де `<тіло>` — це той самий вкладений `if`, але з `count Tree` заміненим на `neighbours`.

### Вправа 4: написати правило з нуля

**Setup:** уявіть просту модель: клітинка типу `Cell` має поле `energy` (0..1). Кожен крок енергія зменшується на 0.01 (метаболізм), збільшується на 0.05 за кожного сусіда у стані `Cell`, і обмежується у [0, 1]. Якщо енергія впала нижче 0.1 — клітинка переходить у стан `Dead`. Інакше залишається `Cell`.

**Завдання:** напишіть `let cell Cell = ...`, використовуючи `let`-ланцюжок для проміжних обчислень: метаболізм, бонус від сусідів, нова енергія, перехід стану.

**Підказка:** структура має виглядати приблизно так:

```swl
let cell Cell =
  let e        = field energy in
  let metab    = ... in
  let bonus    = ... in
  let new_e    = clamp ... in
  if new_e < 0.1 then Dead
  else seq { set energy = new_e }
```

(`count Cell` дасть кількість сусідів у стані `Cell`. `clamp x 0.0 1.0` обмежить значення.)

## Підсумок

`let x = 5 in expr` — це не присвоєння. Це вираз, який каже "значенням цілого є значення `expr`, у якому `x` означає `5`". Це не модифікує пам'ять, не створює "змінну", не виконує "інструкцію". Це просто спосіб назвати підвираз.

Звідси виростають всі властивості, які роблять функціональний код приємним: lexical scoping (область видимості видно з тексту), immutability (binding не можна перезаписати), shadowing (можна "перевикористати" ім'я через нове binding, не модифікуючи старе).

`let`-ланцюжок (як у Gray-Scott прикладі) — це не сім інструкцій, а одне велике обчислення з сімома проміжними іменами. Це дозволяє писати математичні формули кроками, як у підручнику, без втрати продуктивності і без втрати структури.

Якщо ви до цього писали Python без перезапису локальних змінних, ви вже інтуїтивно знаєте, що таке `let`. Тепер у вас є явна назва і явний синтаксис для цього патерну. І коли ви наступного разу побачите сім `let`-ів підряд у Swarmlet або OCaml коді — ви прочитаєте їх як рецепт, а не як головоломку.
