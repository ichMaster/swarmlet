# Чистота і побічні ефекти у Swarmlet

Якщо ви прийшли з імперативного світу — Java, Python, Go, C# — слово "чиста функція" звучить як щось зі студентського курсу теорії: красива абстракція, яку приємно обговорювати, але яка швидко розчиняється у реальному коді з логером, базою даних і HTTP-клієнтом. Цей документ — спроба показати, що "чистота" — це не релігійна догма, а конкретний інженерний інструмент, який має конкретні наслідки для тестування, рефакторингу і розуміння коду. І що, як не дивно, чи не найкраще місце, щоб цю ідею відчути на практиці — це интерпретатор маленької мови симуляцій.

Swarmlet написаний так, що **майже все є чистим**. Парсер, аналізатор, evaluator виразів — усе це функції без побічних ефектів. Єдині компоненти, які мають побічні ефекти, — це `World.step()` (мутує внутрішній стан, але робить це детерміновано) і CLI-шар (читає з диска, пише на диск, друкує у термінал). Ця архітектура — не випадковість, і не "так вийшло". Це навмисний дизайн, який має ім'я: **functional core, imperative shell**.

Якщо ви будете розуміти, *чому* саме так, ви будете розуміти не тільки Swarmlet, але і будь-який добре спроектований інтерпретатор, компілятор, або movement engine у грі — бо всі вони побудовані за тим самим принципом.

Цей документ — для старшого імперативного інженера, який хоче зрозуміти, що насправді означає "функціональна архітектура" — без академічного жаргону, з конкретними прикладами з реального коду Swarmlet.

## Що таке чиста функція насправді

Визначення коротке, але кожне слово важливе. Функція є **чистою (pure)**, якщо виконуються дві умови:

1. **Однаковий вхід — однаковий вихід.** Скільки б разів ви не викликали `f(x)`, з тим самим `x` ви завжди отримаєте той самий результат. Сьогодні, завтра, у пам'яті іншого процесу, на іншому континенті — байдуже. Результат залежить **виключно** від аргументів.

2. **Жодних побічних ефектів (no side effects).** Виклик функції не змінює нічого, крім обчислення результату. Не пише у файл, не змінює глобальну змінну, не модифікує переданий об'єкт, не друкує у консоль, не надсилає мережевий запит, не питає поточний час.

Якщо обидві умови виконуються, як **наслідок** виходить третя властивість, яка має красиве ім'я — **referential transparency** (референтна прозорість): у будь-якому місці програми ви можете замінити виклик `f(x)` на його результат, і програма поведе себе абсолютно так само. Це не ще одне визначення чистоти — це її практичний тест. Ми ще повернемося до нього окремо.

### Чому два пункти, а не один

Тут є тонкість, яка зразу не очевидна. Здавалося б, "немає побічних ефектів" вже означає "однаковий вхід — однаковий вихід". Якщо функція нічого не міняє і нічого зовні не читає — звідки б взялася неоднаковість?

Але "немає побічних ефектів" — це про *запис*. А "однаковий вхід — однаковий вихід" — це насамперед про *читання*. Уявіть функцію, яка не змінює нічого зовні, але читає поточний час:

```python
def greet(name):
    hour = datetime.now().hour  # читає зовнішній стан
    if hour < 12:
        return f"Good morning, {name}"
    return f"Good evening, {name}"
```

Жодного запису назовні — функція тільки повертає рядок. Але вона **не чиста**, бо `greet("Alice")` о 11:59 і `greet("Alice")` о 12:01 повертають різні результати. Вхід однаковий — вихід різний. Тому потрібна перша умова окремо: вона ловить *читання* зовнішнього стану.

Друга умова, своєю чергою, ловить *запис*. Функція може повертати завжди одне й те саме, але по дорозі друкувати у консоль. Технічно для викликача це неважливо — він отримає той самий результат. Але кожен виклик залишає слід. Це порушує принцип "виклик можна замінити результатом" — бо результат не друкує, а виклик друкує.

### Чому це не просто академічна гра

Імперативний інженер часто питає: "ну і що? Якщо моя функція час від часу пише у лог, що в цьому страшного?" Нічого. Поки не доводиться:

- **Тестувати** її без mock-ів і без файлової системи
- **Запускати** її у двох потоках і дивуватися, чому інколи виходить дивне
- **Кешувати** її результат, бо вона повільна
- **Розуміти**, що саме відбувається у складній функції з 10 викликами всередині

Кожна з цих ситуацій — це момент, коли чистота починає окуплятися. Тому ця властивість — не "правильно vs неправильно", а "коли і де".

## Приклади чистих і нечистих функцій

Подивимось на конкретику. Ось чисті функції з реального світу:

```python
def add(a, b):
    return a + b

def length(xs):
    return len(xs)

def square(x):
    return x * x

def absolute(x):
    if x < 0:
        return -x
    return x

def parse_int(s):
    return int(s)  # кидає виняток на поганому вході — все одно чиста
```

Останній приклад може здивувати. Хіба виняток — не побічний ефект? Технічно — ні. Виняток — це інший спосіб повернення з функції; він не змінює зовнішнього стану. Якщо `parse_int("hello")` завжди кидає той самий `ValueError`, функція чиста. Це поведінка детермінована.

А ось нечисті:

```python
def random_int():
    return random.randint(0, 100)  # вхід порожній, вихід різний

def print_hello():
    print("hello")  # пише у stdout
    return None

def write_file(path, data):
    with open(path, "w") as f:
        f.write(data)  # пише на диск
    return None

def now():
    return time.time()  # читає системний годинник

def append_to_list(xs, item):
    xs.append(item)  # мутує переданий список
    return xs
```

Останній приклад особливо підступний. Він **виглядає** як функція, що повертає список, але насправді він мутує переданий аргумент. Викликач передав `xs`, отримав `xs` назад — але *той самий* `xs`, тепер з лишнім елементом. Якщо у викликача є інший reference на цей же список, він тепер бачить мутацію, якої не очікував. Це класичне джерело багів у Python і JavaScript.

Чисті функції з кодової бази Swarmlet:

- `_type_tag(v)` у `swarmlet/eval.py` — приймає Python-значення, повертає string-тег. Жодного зовнішнього стану.
- `eval_expr(expr, env, ctx)` у `swarmlet/eval.py` — приймає AST вузол, environment і контекст; повертає значення. Не міняє нічого з переданого.
- `lex(source)` у `swarmlet/lexer.py` — приймає вихідний код, повертає список токенів.
- `parse(tokens)` у `swarmlet/parser.py` — приймає токени, повертає AST.

Усі ці функції можна викликати скільки завгодно разів, у будь-якому порядку, з будь-якого потоку — і вони завжди дадуть той самий результат. Якщо ви знайшли баг у `eval_expr`, ви можете відтворити його, просто передавши той самий вхід. Не потрібно відтворювати "стан системи", не потрібно ставити mock-и, не потрібно турбуватися про порядок виклику.

## Тест чистоти: підстановка

На практиці перевірити чистоту простіше за визначення. Запитайте себе: **якби я замінив виклик функції на його результат, чи поводилася б програма ідентично?**

Дивіться:

```python
result = add(2, 3)
print(result)
# еквівалентно:
print(5)
```

Працює. `add` — чиста.

А ось:

```python
result = print_hello()
print(result)
# спробуємо замінити:
print(None)
```

**Не еквівалентно.** Перший варіант друкує "hello" і потім "None", другий тільки "None". Замінити виклик на результат не вийшло — побічний ефект "втратився". Отже, `print_hello` — нечиста.

Цей тест дуже корисний на code review. Якщо ви бачите функцію і не можете однозначно сказати, чи можна замінити її виклик на результат, — це сигнал: чистота "розмита", і десь усередині може бути прихований ефект.

Цей тест працює і для бібліотечних функцій. `numpy.sum(arr)` — чиста (можна замінити виклик на число). `arr.append(x)` — нечиста (заміна на `None`, який повертає метод, не дасть того ж ефекту, бо список не зміниться). `dict[key]` — чиста (детерміновано повертає значення). `dict[key] = value` — нечиста (виконує запис, повертає `None`).

## Класифікація модулів Swarmlet

Тепер головне. Подивимось на сам Swarmlet і пройдемо по кожному модулю — чи він чистий, чи "контрольовано-нечистий", чи відверто нечистий. Це не теоретична вправа: розуміючи цю карту, ви розумієте, де у системі "ядро" і де "оболонка".

### Чисті модулі

**`swarmlet/lexer.py`**. Функція `lex(source: str) -> List[Token]` приймає рядок з вихідним кодом і повертає список токенів. Жодного IO, жодних глобальних змінних, жодних мутацій вхідного рядка (рядки у Python імутабельні, але навіть якби й ні). Ви можете прогнати один і той самий source через `lex` мільйон разів — отримаєте мільйон ідентичних списків.

**`swarmlet/parser.py`**. Парсер бере список токенів і будує AST. Внутрішньо у парсера є позиційний індекс (mutating cursor), але це **локальна** мутація — вона не виходить за межі виклику. Зовні парсер чистий: same input, same output.

**`swarmlet/ast.py`**. Власне AST — це frozen dataclasses. Це навіть не функції, а структури даних, але вони **імутабельні**, що робить їх ідеальним матеріалом для чистих обчислень. Ви не можете випадково зіпсувати AST вузол, бо `frozen=True` забороняє запис.

**`swarmlet/analyzer.py`**. Аналізатор створює `Analyzer()`-інстанс, накопичує помилки у `self.errors`, і кидає виняток у кінці. Так, всередині є мутація `self.errors.append(...)`. Але — і це ключовий момент — публічний контракт `analyze(program) -> None | raises` чистий: один і той самий вхід дасть одну й ту саму поведінку. Внутрішня мутація локальна; вона не "тікає" назовні. Для зовнішнього коду analyzer — чорна скринька, яка детерміновано приймає AST і повертає або тишу, або `SwarmletStaticError`.

**`swarmlet/eval.py`**. Тут найцікавіше. `eval_expr(expr, env, ctx)` приймає вираз AST, environment і evaluation context — і повертає **значення**. Він *не* записує нічого у world. Він *не* модифікує environment (зробивши `let x = 1 in body`, evaluator створює *нове* environment, в якому є `x`, і обчислює `body` у ньому — старе залишається недоторканим). Action evaluator (`eval_action`) теж нічого не виконує "по-справжньому": він повертає `Intent` — описову структуру намірів. Це принципово інша річ — про неї детально у [intent-pattern-explained.md](06-intent-pattern-explained.md).

### Контрольовано-нечисті модулі

**`swarmlet/engine.py`**. Ось де лежить мутація. Подивіться на конструктор `World`:

```python
def __init__(self, program, seed=0, params=None):
    self.program = program
    self.seed = seed
    self.rng = np.random.default_rng(seed)
    self.tick_count = 0
    # ...
    self.states = np.zeros((self.height, self.width), dtype=np.int32)
```

`World` тримає `self.states`, `self.cell_fields`, `self.agents`, `self.tick_count`. На кожному `step()` ці поля **мутуються**: створюється нова snapshot стану, потім вона записується назад у `self.states`. Це класичний імперативний код.

Але тут є тонкість, яка все міняє: **публічна поведінка `World` детермінована**. Якщо ви створите два `World` з одним і тим самим program і seed і прокрутите їх однакову кількість тіків — отримаєте однакові snapshot-и. Внутрішня мутація обмежена і контрольована. Зовнішній спостерігач не бачить різниці між "World, який мутує себе" і "World, який повертає новий World". Ця властивість має ім'я: **observational purity** — функція може мати ефекти всередині, але якщо назовні її поведінка чиста, для більшості практичних цілей вона еквівалентна чистій.

Чому не зробили все по-справжньому імутабельно? Бо це гра з продуктивністю. Копіювати numpy-масив `1000x1000` на кожному тіку — повільно. Тримати один масив і переписувати його in-place — швидко. Гарантія детермінізму утримується через дисципліну: всі мутації відбуваються **тільки** у `step()`, і завжди в одному й тому самому порядку. Детально про це — у [tick-as-snapshot-transformation.md](07-tick-as-snapshot-transformation.md).

**`swarmlet/snapshot.py`**. Має дві функції: одна читає стан world (чиста — просто збирає поточний стан у JSON-сумісний словник), інша **пише** його у файл (нечиста — `open(path, "w")` і `f.write(...)`). Module — змішаний. Внутрішня функція `_world_to_dict` чиста; обгортки `write_jsonl` і `write_npz` — ні.

### Нечисті модулі

**`swarmlet/cli.py`**. CLI — це край системи, де імперативний світ зустрічає функціональне ядро. Тут читається файл (`open(args.file).read()`), друкуються повідомлення про прогрес (`print(...)`), пишуться вихідні snapshot-и (`write_jsonl(...)`), реєструється системний час (`time.time()`), завершується процес (`sys.exit(1)`). Жодна з цих операцій не може бути чистою за визначенням — їхня вся суть саме в тому, щоб взаємодіяти зі світом за межами Python-процесу.

Це **і є правильно**. Кожна програма десь має взаємодіяти з зовнішнім світом — інакше її ніхто не побачить. Питання не в тому, як цього уникнути, а в тому, *де провести межу*.

## Functional core, imperative shell

Цей принцип популяризував **Gary Bernhardt** у відомій доповіді 2012 року "Boundaries". Ідея проста: усю логіку, яка приймає рішення, обчислює результати, перетворює дані — пишіть як чисті функції. Усе IO, мутацію, взаємодію зі світом — винесіть на тонку оболонку навколо ядра.

Архітектурно це виглядає як концентричні кола:

```
   +-----------------------------+
   |       imperative shell      |   <- CLI, IO, sys, time, файли
   |                             |
   |   +---------------------+   |
   |   |   functional core   |   |   <- lexer, parser, analyzer,
   |   |                     |   |       eval, ast (чисті)
   |   +---------------------+   |
   |                             |
   +-----------------------------+
```

`engine.py` — це проміжний шар: всередині мутує, ззовні чистий. У термінології Bernhardt — це теж частина ядра, бо її контракт детермінований.

Чому це працює? Бо **тестування і міркування про чисте ядро тривіальне**, а нечиста оболонка — *тонка* і має одну роботу: переганяти дані з світу в ядро і назад. Якщо ви добре спроектували розподіл, 90% коду тестується без жодного mock-у, без `tempfile`, без monkey-patching `time.time()`. А решта 10% — настільки прямолінійна, що тести майже не потрібні.

У Swarmlet це виглядає так:

- `lex`, `parse`, `analyze`, `eval_expr`, `eval_action` — пишуть unit-тести з літеральними входами і літеральними очікуваннями. Жодних моків. Жодних temp-файлів. Жодного "перед тестом створіть стан".
- `World.step()` — пишуть unit-тести, які створюють World з малою програмою, виконують `step()`, перевіряють фінальний стан. Більше нічого не потрібно — World детермінований, тест відтворюваний.
- `cli.py` — інтеграційні тести через `subprocess.run`, які перевіряють, що "swarmlet run example.swl -t 10 -o out.jsonl" створює очікуваний `out.jsonl`. Тут уже є IO, але тестів цього рівня одиниці.

Майже всі баги ловляться у нижньому шарі, бо нижній шар і є місцем, де живе складність. Оболонка — нудна.

## Чому чистота важлива на практиці

П'ять конкретних виграшів — не теоретичних, а тих, які ви відчуєте відразу.

### 1. Тестування

Чиста функція — це функція, для якої тест має вид `assert f(input) == expected`. Все. Не потрібен `setUp`, не потрібен `tearDown`, не потрібен `pytest.fixture`. Якщо ви змогли передати вхід — ви змогли написати тест.

```python
def test_lex_simple_let():
    tokens = lex("let x = 5")
    assert tokens[0].kind == "LET"
    assert tokens[1].kind == "IDENT"
    assert tokens[2].kind == "EQ"
    assert tokens[3].value == 5
```

Якщо `lex` був би нечистий — наприклад, читав з якогось файлу конфігурації, — цей тест вже потребував би налаштування, mock-а файлової системи, контролю над `os.environ`, всього того болота, яке ми всі знаємо.

### 2. Міркування про код

Чисту функцію можна **локально** зрозуміти. Прочитав сигнатуру, прочитав тіло — все, ви знаєте всю поведінку. Не треба думати "а що, якщо вона викликається в іншому потоці", "а в якому стані буде глобальна змінна на момент виклику", "а чи не змінив хтось файл за моєю спиною".

Нечисту функцію треба розуміти **глобально**: треба знати весь контекст, у якому вона працює. Це принципова різниця у когнітивному навантаженні.

### 3. Рефакторинг

Якщо функція чиста, ви можете її витягнути, перенести у інший файл, перейменувати, інлайнити — все це безпечно. Семантика не зміниться. Якщо нечиста — кожен такий рух потенційно ламає неявні залежності.

### 4. Memoization (кешування)

Чисту функцію можна кешувати тривіально:

```python
from functools import lru_cache

@lru_cache(maxsize=None)
def fib(n):
    if n < 2:
        return n
    return fib(n-1) + fib(n-2)
```

Працює, бо `fib(10)` завжди `55`, незалежно від того, коли і де викликали. Спробуйте кешувати `random.randint(1, 100)` — отримаєте перше випадкове число назавжди, що, очевидно, не те, що ви хотіли.

У Swarmlet `eval_expr` теоретично можна було б memoize-ити для повторюваних виразів — і це було б коректно, бо функція чиста. На практиці накладні витрати на хешування AST вузлів роблять це сумнівним, але можливість *є*. З нечистою функцією її не було б у принципі.

### 5. Паралелізація

Чисті функції можна викликати паралельно без жодних блокувань і синхронізації. Немає спільного стану — немає race condition. Немає race condition — немає мьютексів. Немає мьютексів — немає deadlock-ів.

`eval_expr` для незалежних клітин гіпотетично можна було б розпаралелити — кожна клітина обчислюється сама по собі, не залежить від інших. (У Swarmlet це не зроблено, бо для типових розмірів grid-ів накладні витрати threading-у переважують виграш — але це питання продуктивності, а не семантики.)

Спробуйте розпаралелити `World.step()`, який мутує `self.states`. Доведеться або копіювати масиви, або вводити блокування. Складність зростає експоненційно.

## Контрольована нечистота: seeded RNG

Тепер цікавий випадок. Що робити, коли вам *потрібна* випадковість? Чисті функції за визначенням детерміновані; випадковість за визначенням ні. Здається, парадокс.

Розв'язок — **передавайте джерело випадковості як аргумент**. Це робить функцію "однакового входу — однакового виходу" знов чистою:

```python
# нечиста — кожен виклик різний
def random_position():
    return (random.randint(0, 100), random.randint(0, 100))

# чиста — вхід rng визначає вихід
def random_position(rng):
    return (rng.integers(0, 100), rng.integers(0, 100))
```

У другому варіанті, якщо ви передасте однаково ініціалізований `rng`, ви отримаєте однаковий результат. Випадковість тепер — частина входу, а не "магічна" властивість функції.

У Swarmlet ця ідея реалізована через **seeded RNG**. World створюється з `seed: int`:

```python
self.rng = np.random.default_rng(seed)
```

І всюди, де треба випадковість — порядок обходу агентів, випадковий вибір сусіда, випадкове розміщення нових агентів, — використовується `self.rng`, а не модульний `numpy.random`. Це означає, що:

- Запустіть World з seed=42, прокрутіть 1000 тіків — отримаєте конкретну послідовність snapshot-ів.
- Запустіть знову з seed=42 — отримаєте *ту саму* послідовність. До останнього біта.
- Поміняйте seed=43 — отримаєте іншу послідовність, але теж відтворювану.

Це і є "контрольована нечистота": формально RNG нечистий (його стан змінюється на кожному виклику), але *розмов*но детермінований щодо seed-у. Тести, налагодження, відтворення багів — все стає тривіальним. Деталі — у `tests/test_determinism.py`.

## OCaml і Haskell: дві школи функціонального програмування

Цей розділ — для контексту. Якщо ви ніколи не писали на функціональних мовах, корисно знати, що навіть у "функціональному світі" існує два дуже різні підходи до чистоти.

### OCaml: чистота за домовленістю

OCaml — функціональна мова з ML-родини (Swarmlet синтаксично багато бере звідти: `let`, `match ... with`, immutable за замовчуванням). Але OCaml **не примушує** функції бути чистими. Можна писати:

```ocaml
let greet name =
  print_endline "hello";
  "Hi, " ^ name
```

`print_endline` — функція з типом `string -> unit`, яка друкує і повертає `()`. Нічого не сигналізує про побічний ефект — тип `unit` такий же, як у будь-якої функції, що "нічого не повертає". Якщо ви хочете дізнатися, чи `greet` чиста — читайте її код. OCaml вам не допоможе.

Це зручно — ви можете швидко написати IO там, де потрібно, без церемоній. І це небезпечно — у великій кодовій базі легко загубити, де ховаються ефекти. Конвенція ком'юніті OCaml — "чисте, поки можеш; нечисте, коли треба", — але це саме конвенція, не закон мови.

Swarmlet (інтерпретатор Python) живе у тій самій парадигмі. Python нічого не знає про чистоту. Дисципліна — на нас.

### Haskell: чистота, вшита у систему типів

Haskell пішов інакше. Тут будь-який побічний ефект **обов'язково** видно у типі функції. Хочете надрукувати у консоль? Ваша функція тепер має тип, який містить `IO`:

```haskell
greet :: String -> IO String
greet name = do
  putStrLn "hello"
  return ("Hi, " ++ name)
```

`String -> IO String` — це не просто "приймає рядок, повертає рядок". Це "приймає рядок, повертає **обчислення в IO**, яке коли виконається, дасть рядок". `IO` — це **monad** (модний термін, який тут означає "тип-обгортка з певним інтерфейсом"). Будь-яка функція, яка торкається IO — файлів, мережі, стану, часу, — має `IO` у своєму типі.

Що це дає? Ви на рівні типу бачите, які функції чисті, а які ні:

```haskell
add :: Int -> Int -> Int                        -- чиста: типи прості
greet :: String -> IO String                    -- нечиста: IO у типі
parseConfig :: String -> Either Error Config    -- чиста: ніякого IO
readConfig :: FilePath -> IO Config             -- нечиста: читає диск
```

Компілятор не дозволить вам "випадково" викликати IO-функцію зсередини чистої. Спробуєте — отримаєте помилку типізації. Чистота вшита у мову.

Це жорстко. Це може дратувати, коли ви хочете швидко надрукувати щось для дебагу. Але це залізно гарантує: **якщо тип функції без `IO` — функція чиста. Точка.** Жодних "майже" і "переважно".

### Що з цього важливо для Swarmlet

Swarmlet написаний на Python, який, як і OCaml, нічого не примушує. Але архітектура свідомо відображає Haskell-підхід: чітка межа між чистим ядром і нечистою оболонкою. Просто межу проводимо ми, інженери, читаючи код, а не компілятор, читаючи типи.

Якщо вам цікаво, як ця межа виглядає у мові, де вона *обов'язкова* — подивіться кілька годин Haskell. Ви не пошкодуєте. А коли повернетесь у Python чи Java, у вас з'явиться новий "детектор": ви будете *бачити* функції, які потайки роблять IO, у місцях, де воно зайве.

## Поширені пастки

Чотири місця, де чистота губиться непомітно навіть у досвідчених інженерів.

### Пастка 1: мутація переданих аргументів

```python
def process(items):
    items.sort()         # мутує переданий список
    return items[:10]
```

Ця функція виглядає як "повертає перші 10 відсортованих", але насправді ще й сортує оригінал. Викликач, який передав `items`, тепер має відсортований `items` — навіть якщо він цього не очікував. Правильно:

```python
def process(items):
    return sorted(items)[:10]   # sorted повертає нову копію
```

### Пастка 2: приховане читання глобального стану

```python
DEBUG = False

def compute(x):
    if DEBUG:
        print(f"computing {x}")
    return x * 2
```

`compute(5)` зазвичай повертає `10`. Але якщо хтось десь поставив `DEBUG = True` — функція починає друкувати у stdout. Поведінка функції залежить від глобальної змінної. Це нечисто, навіть якщо ефект "невинний".

### Пастка 3: ефекти у конструкторі

```python
class Config:
    def __init__(self, path):
        self.data = json.loads(open(path).read())   # IO у конструкторі
```

`Config("foo.json")` робить читання файлу. Якщо файл змінився між двома викликами — ви отримаєте різні `Config`. Якщо файл недоступний — конструктор кидає виняток. Конструктор виглядає як "створи об'єкт", але насправді він є нечистою операцією, прихованою за синтаксисом конструктора.

Кращий патерн:

```python
class Config:
    def __init__(self, data):
        self.data = data

    @classmethod
    def from_file(cls, path):
        return cls(json.loads(open(path).read()))
```

Тепер `Config(data)` — чисте, `Config.from_file(path)` — явно нечисте. Ви бачите, де IO, по імені виклику.

### Пастка 4: "просто логування"

```python
def add(a, b):
    logger.debug(f"adding {a} and {b}")
    return a + b
```

"Це ж тільки лог". Так, але тепер `add(2, 3)` залежно від logger config може робити IO у файл, у сокет, у Sentry. Тестувати таку функцію стає важче (треба ставити mock на logger). Кешувати її — все ще можна (logger ефект "невидимий" для результату), але якщо ви забули — це маленька мина.

Альтернатива — логування на рівні викликача, або структуровані результати з полем "trace info", які obviously різні від основного значення. Але часто прагматика перемагає, і ми просто погоджуємось: "цей лог — допустимий маленький ефект, з яким миримось".

Головне — *усвідомлювати*, що це маленький ефект. Не давати йому вилитись у щось більше.

## Вправи

Чотири задачі, що допоможуть закріпити інтуїцію. Спробуйте їх перш ніж читати підказки.

### Вправа 1: класифікуйте функції

Дано п'ять функцій. Для кожної визначте — чиста, контрольовано-нечиста, чи відверто нечиста. Поясніть.

```python
def f1(x, y):
    return (x + y) ** 2

def f2(items, predicate):
    return [x for x in items if predicate(x)]

class Counter:
    def __init__(self):
        self.n = 0
    def f3(self):
        self.n += 1
        return self.n

def f4(name):
    os.environ["LAST_NAME"] = name
    return name.upper()

def f5(rng):
    return rng.integers(0, 100)
```

**Підказка.** Проведіть тест підстановки: якби викликач замінив виклик на результат, чи поводилася б програма так само? Для `f5` подумайте про "контракт `rng`": якщо `rng` мутується всередині, то функція мутує переданий аргумент — це нечисто *локально*. Але якщо ми домовились, що мета `rng` саме у цьому, ефект контрольований.

### Вправа 2: зробіть чистим

Перепишіть так, щоб функція стала чистою. Що довелось змінити у сигнатурі?

```python
import time

def time_it(fn, *args):
    start = time.time()
    result = fn(*args)
    elapsed = time.time() - start
    print(f"took {elapsed}s")
    return result
```

**Підказка.** Тут два побічні ефекти: читання поточного часу і друк. Перший можна "винести у параметр" — передавати `clock` як функцію. Другий — повертати elapsed разом з результатом, а друк зробити в іншому місці. Сигнатура зміниться: повертатиме tuple `(result, elapsed)`, а друк буде відповідальністю викликача.

### Вправа 3: знайдіть ефект у Swarmlet

Відкрийте `swarmlet/eval.py`. Прочитайте функцію `eval_expr`. Чи знайдете ви хоч один побічний ефект? Якщо так — який саме і чи виходить він за межі виклику?

**Підказка.** Дивіться на: створення environment-ів (нові словники чи мутація?), накопичення помилок, виклики builtin-ів. Якщо знаходите щось підозріле — задайте питання: "чи бачить це викликач?". Якщо ні — це "локальна" нечистота, яка з боку ззовні не порушує чистоту.

### Вправа 4: спроектуйте чисту обгортку

Уявіть, що `World.step()` все ще мутує внутрішнє, але ви хочете надати чисту обгортку — функцію `pure_step(world_snapshot, program, seed, tick) -> new_world_snapshot`. Як би вона виглядала? Що таке `world_snapshot` тут?

**Підказка.** Snapshot — це frozen dataclass з усім, що описує стан: `states` (numpy array), `cell_fields` (dict), `agents` (tuple), `tick_count`, `rng_state` (так, навіть стан RNG треба передати, інакше неможливо детермінувати наступний). Ваша `pure_step` створює тимчасовий World з цього snapshot-у, виконує `step()`, повертає новий snapshot. Це перетворює engine.py на формально чисту функцію — ціною копіювання масивів. Питання: за яких обставин ця ціна виправдана?

## Підсумок

Чистота — це не релігія "FP проти всіх". Це інженерний інструмент, у якого є чіткі плюси (тестування, рефакторинг, паралелізація) і чіткі мінуси (швидкість, незручність для IO, потреба явно передавати стан). Завдання дизайнера системи — провести межу так, щоб 90% коду опинилося у чистій частині, а 10% — у тонкій нечистій оболонці.

Swarmlet — приклад такої архітектури. Lexer, parser, analyzer, evaluator — чисті функції, які тестуються тривіально. Engine — контрольовано-нечистий: всередині мутує, ззовні детермінований. CLI і snapshot writers — нечисті, але тонкі: їхня роль — переганяти дані з файлу в ядро і назад.

Якщо ви будете тримати цю карту в голові, коли пишете свій код — навіть на мові, яка нічого не примушує, як Python чи Java, — ви побачите, як код стає простішим, тести швидшими, баги передбачуванішими. А коли ви глянете на Haskell і його `IO`-monad, ви впізнаєте у ньому не "академічну надмірність", а просто *інший спосіб* провести ту саму межу — машинно перевірений замість дисциплінарного.

Чистота — це межа. Питання тільки в тому, де ви її проведете і хто її буде стерегти.
