# Tick як трансформація snapshot-ів

Якщо ви приходите з імперативного світу, слово "tick" викликає у голові цілком конкретну картинку. Є об'єкт `World`, у нього є метод `update()`, цей метод бігає по сітці у двох вкладених циклах і **мутує** клітинки на місці. Спочатку оновили клітинку (0, 0), потім (0, 1), потім (0, 2), і так далі. Це класичний game loop. NetLogo, Unity, Conway's Game of Life у будь-якому туторіалі на JavaScript — усі вони так і виглядають.

Цей документ — про те, чому в Swarmlet модель **інша**, чому "tick" тут — це **чиста функція над snapshot-ами**, а не процедура, що щось мутує. Ми пройдемо концепцію згори донизу: що таке snapshot, чому tick — це `Snapshot -> Snapshot`, як cell phase і agent phase складаються в одну композицію, що все це дає на практиці, і де у реальному коді є невеличкий "чит", який варто чесно визнати.

Документ розрахований на досвідченого імперативного інженера, який вивчає функційне програмування. Якщо ви вже комфортно почуваєтеся з пайплайнами в OCaml або Elixir — деякі речі будуть очевидними, але ми все одно пройдемо їх повільно, бо tick є саме тим місцем, де FP-шні ідеї стикаються з продуктивністю і чесністю.

Перед прочитанням варто пробігти очима [SPEC.md секція 2](../../specification/Swarmlet-SPEC.md#2-architecture) — там лаконічно описана модель виконання. Цей текст — її розгорнуте пояснення з ухилом у "чому саме так".

## Звичайна модель: tick як імперативна процедура

Спочатку подивімося, як зазвичай виглядає tick у "наївному" симуляторі. Ось приклад на Python — реалізація лісової пожежі:

```python
class World:
    def __init__(self, width, height):
        self.cells = [[Empty() for _ in range(width)] for _ in range(height)]

    def tick(self):
        for y in range(self.height):
            for x in range(self.width):
                cell = self.cells[y][x]
                if isinstance(cell, Burning):
                    self.cells[y][x] = Ash()
                elif isinstance(cell, Tree):
                    if any_neighbor_burning(self.cells, x, y):
                        self.cells[y][x] = Burning()
```

Виглядає просто. Але всередині є тонкий і дуже неприємний баг. Коли ми обходимо клітинки зверху вниз, зліва направо, і **мутуємо їх на місці**, то наступна клітинка бачить **частково оновлений світ**. Якщо клітинка (5, 5) загорілася на цьому тіку, то клітинка (5, 6), яку ми оновлюємо наступною, побачить її вже як `Burning` — і теж загориться **у той самий тік**. Це **несинхронне оновлення**, і воно дає не дискретну хвилю пожежі, а вибух у бік сходу-півдня. Поведінка стає залежною від порядку обходу.

Класичний фікс — мати **дві сітки**: `current` і `next`. Читаєш з `current`, пишеш у `next`, наприкінці тіку міняєш їх місцями. Це вже дуже близько до того, що робить Swarmlet, тільки Swarmlet піднімає цю ідею до рівня семантики мови, а не лишає її як деталь реалізації, яку легко зіпсувати.

Друга проблема імперативної моделі — **тестування**. Як перевірити, що функція `tick` працює коректно для конкретної конфігурації? Доводиться:

1. Створити цілий об'єкт `World`.
2. Вручну встановити стан клітинок.
3. Викликати `tick()`.
4. Прочитати назад стан клітинок і порівняти.

Усе це лежить на одному великому об'єкті з купою прихованого стану. Ви не можете просто сказати "ось вхід, ось очікуваний вихід, перевір". Об'єкт стає "лопатою для тестів" — ви постійно копаєте всередину нього, щоб витягти або підкласти щось.

Третя проблема — **відлагодження**. Симуляція пройшла 1000 тіків, на 847-му щось зламалося. Як подивитися, що було на 846-му? У звичайній мутабельній моделі — ніяк. Стан переписаний. Якщо ви хочете "перемотати назад", треба було логувати кожен крок окремо, у форматі, з якого можна реконструювати стан. Зазвичай ніхто цього не робить, бо дорого.

Усі ці три біди — несинхронність, тестованість і відлагодженість — вирішуються однією зміною мислення. Ось вона.

## Інша модель: snapshot як значення, tick як функція

Уявіть, що **весь стан світу** — сітка станів, усі поля, усі агенти — це одне велике **immutable значення**. Назвімо його `Snapshot`. Це не об'єкт із методами. Це структура даних. Картинка світу у конкретний момент часу.

```python
@dataclass(frozen=True)
class Snapshot:
    states: np.ndarray         # 2D масив (H, W) станів клітинок
    fields: Dict[str, np.ndarray]   # ім'я поля -> 2D масив значень
    agents: Tuple[AgentRecord, ...] # immutable tuple агентів
    tick_count: int
```

Snapshot не знає, як змінитися. Він просто **є**. Він описує: "ось такі клітинки, ось такі поля, ось такі агенти, на тіку номер N".

Тепер означимо tick. У Swarmlet tick — це **чиста функція** з підписом:

```
tick : Snapshot -> Snapshot
```

Береш snapshot, повертаєш новий snapshot. Старий — недоторкуваний. Між ними немає shared mutable state. Якщо ви захочете порівняти `before` і `after` — обидва ще тут, як два самостійних значення.

Це не просто "інша архітектура". Це **інша математична природа**. У імперативній моделі симуляція — це послідовність побічних ефектів. У функційній — це послідовність значень, з'єднаних чистою функцією:

```
s_0  --tick-->  s_1  --tick-->  s_2  --tick-->  s_3  ...
```

Ця стрічка snapshot-ів — і є симуляція. Вона існує цілком, як математичний об'єкт. Кожен `s_n` — повний опис світу на тіку `n`. Ви можете взяти будь-який з них і робити з ним що завгодно: рендерити, аналізувати, серіалізувати, відкинути.

Це як різниця між "я виконую алгоритм" і "я обчислюю функцію". Імперативно — я роблю кроки, що ведуть до результату. Функційно — я визначаю відображення з входу у вихід, і весь алгоритм — це просто **спосіб обчислити це відображення**, а не сама його суть.

### Чому immutable

Найболючіше питання для імперативного інженера: "а як же продуктивність? Кожен тік ми створюємо нову копію всього світу?". Чесна відповідь — так, концептуально саме це й відбувається. Але насправді є дві оптимізації, які роблять це дешевим:

1. **Structural sharing.** Якщо нова версія snapshot відрізняється від старої лише у частині клітинок, можна не копіювати все — лише ту частину, що змінилася. Persistent data structures (Clojure, Haskell, Scala) роблять це автоматично.
2. **Копіювання з мутацією під капотом.** Ви говорите про immutable значення на рівні **семантики**, але реалізація може фізично мутувати буфер, якщо вона гарантує, що ніхто інший його не побачить.

Swarmlet використовує другий підхід. Це і є та сама "невелика чесність", про яку ми поговоримо у відповідній секції нижче. Але концептуально — кожен tick породжує нове значення, попередній snapshot — недоторкуваний.

### Snapshot як цінність сам по собі

Зверніть увагу на одну дивну річ. У імперативній моделі **немає** концепції "snapshot". Стан світу — це набір полів об'єкта `World`, до якого ви маєте доступ у конкретний момент. "Snapshot" виник би тільки якби ви явно скопіювали все у словник — для логування, наприклад. Але це був би побічний продукт, не первинна сутність.

У функційній моделі **snapshot — це первинна сутність**. World — лише обгортка над snapshot-ом, що знає, як з нього отримати наступний. Уся симуляція описується через snapshot-и. Це інверсія, до якої треба звикнути.

Один корисний образ: snapshot — це **кадр фільму**. Симуляція — це послідовність кадрів. Tick — це функція, що з одного кадру обчислює наступний. Самі кадри ні від чого не залежать; вони просто **є**.

## Cell phase як чиста функція

Розберімо tick на дві частини. Перша — cell phase. Її підпис:

```
cell_phase : Snapshot -> Snapshot
```

Бере snapshot, повертає snapshot, у якому всі клітинки оновлені, а агенти не зачеплені.

Що саме вона робить? Для кожної клітинки `(x, y)`:

1. Дивиться на її поточний стан у вхідному snapshot.
2. Знаходить cell rule, яке відповідає цьому стану (`let cell Tree = ...`).
3. Обчислює тіло цього rule, передаючи в нього **вхідний snapshot** як контекст. Cell rule бачить лише старий світ.
4. Результат обчислення — новий стан і нові значення полів для цієї клітинки.
5. Записує результат у **новий, окремий snapshot**.

Ключова річ: **усі обчислення для всіх клітинок дивляться на той самий вхідний snapshot**. Клітинка (5, 6) бачить клітинку (5, 5) у її **старому** стані, навіть якщо ми вже "обробили" (5, 5) у внутрішньому циклі. Бо "обробили" не означає "змінили вхід". Ми просто записали результат у вихід.

Ось це у реальному коді Swarmlet ([engine.py](../swarmlet/engine.py)):

```python
def _cell_phase(self):
    """Evaluate cell rules for every cell, writing into fresh arrays."""
    next_states = self.states.copy()
    next_fields = {fn: arr.copy() for fn, arr in self.fields.items()}

    for y in range(self.height):
        for x in range(self.width):
            state_name = self.state_names[self.states[y, x]]
            body = self._cell_rules.get(state_name) or self._cell_rules.get("_")
            if body is None:
                continue

            ctx = self._make_cell_ctx(x, y)
            # ... обчислюємо body, отримуємо новий стан і нові поля ...
            # ... записуємо у next_states[y, x] і next_fields[fn][y, x] ...

    self.states = next_states
    self.fields = next_fields
```

Зверніть увагу на перший крок: `next_states = self.states.copy()`. Це і є створення нового snapshot-у. Усі обчислення в циклі читають з `self.states` (старий snapshot) і пишуть у `next_states` (новий snapshot). Лише в самому кінці ми робимо "swap" — присвоюємо нові масиви старим іменам.

Це **двофазний commit** на рівні семантики. Поки фаза не завершена, "видимий" світ — старий. Коли завершена — світ скаче в новий стан атомарно.

### Що з цього випливає для cell rules

Раз cell rule — це чиста функція від старого snapshot, то:

- **Порядок обходу клітинок не має значення.** Можете обходити по рядках, по стовпцях, по діагоналях, у випадковому порядку — результат однаковий. Бо жодна клітинка не залежить від уже-обробленої сусідки; усі залежать тільки від вхідного snapshot.
- **Можна паралелізувати.** Тиждень роботи у два потоки на 8 ядер — і ви маєте рівно той самий результат, що й у однопоточному виконанні. Більше про це у секції про паралелізацію.
- **Cell rule не може мати побічних ефектів.** Він не може щось логувати, не може писати у файл, не може мутувати глобальний стан — синтаксично йому це і не дають зробити, бо мова Swarmlet таких конструкцій просто не має.

## Agent phase як чиста функція з intents

Agent phase складніше. Підпис у нього такий самий:

```
agent_phase : Snapshot -> Snapshot
```

Але всередині є нюанс. Агенти можуть **рухатися**. А якщо два агенти хочуть рухатися в одну й ту саму клітинку — є конфлікт. Якщо агент A вбиває агента B на цьому ж тіку — порядок важить. Ці складнощі не розв'язуються тривіальним "обчислити нову позицію кожного агента і записати".

Розв'язання — двоступеневий підхід всередині самої agent phase. Він називається **intent pattern** (детальніше у [intent-pattern-explained.md](06-intent-pattern-explained.md)).

### Крок 1: build phase — збираємо intents

Для кожного агента обчислюємо його rule. Але результат — не нова позиція і не модифікований світ. Результат — **intent record**: декларативний опис того, що агент **хотів би** зробити.

```python
@dataclass
class Intent:
    agent_id: int
    move_dir: int = STAY
    agent_field_writes: Dict[str, float] = field(default_factory=dict)
    cell_field_deposits: List[Tuple[str, float]] = field(default_factory=list)
    spawn_types: List[str] = field(default_factory=list)
    kill_targets: List[str] = field(default_factory=list)
    die: bool = False
```

"Я хочу рухатися на північ. Я хочу записати у своє поле energy значення 5. Я хочу залишити у клітинці поточної позиції pheromone += 1.0. Я хочу заспаунити одного нащадка." Усі ці бажання збираються в `Intent` об'єкт. Жодного впливу на світ ще не сталося.

Це і є хитрість, що зберігає чистоту функції. Build phase читає вхідний snapshot, але не пише в нього. Вихід цієї підфази — список `[(agent, intent), ...]`. Це **значення**, не побічний ефект.

### Крок 2: apply phase — застосовуємо intents атомарно

Тепер у нас є список intents — повний знімок того, що всі агенти хочуть зробити. Ми можемо:

1. **Розв'язати конфлікти.** Якщо двоє хочуть в одну клітинку — кинути жереб (детермінований, через seeded RNG), один іде, інший лишається.
2. **Застосувати в правильному порядку:** рух → запис у поля агентів → депозит у поля клітинок → kill → spawn → death. Цей порядок зафіксований у [SPEC секції 2.3](../../specification/Swarmlet-SPEC.md#2-architecture).
3. **Породити новий snapshot.**

Це і є те, чому "ефекти" з agent phase виглядають як ефекти, але насправді — чисті:

```python
def _agent_phase(self):
    if not self.agents:
        return

    indices = list(range(len(self.agents)))
    self.rng.shuffle(indices)

    intents: List[Tuple[AgentRecord, Intent]] = []

    for idx in indices:
        agent = self.agents[idx]
        if not agent.alive:
            continue
        # ... обчислюємо rule, отримуємо intent ...
        intents.append((agent, intent))

    # Apply intents in order (SPEC section 2.3)
    self._apply_intents(intents)
```

Бачите дві окремі стадії? Перший цикл — це build (збирання intents). Виклик `_apply_intents` — це apply (застосування). Між ними є цілий список intents — **значення**, з якого можна зробити що завгодно: серіалізувати, проаналізувати, переграти. Це і є ключова річ.

### Чому це важливо

Якби ми робили "імперативно" — рух, kill, spawn, deposit одразу всередині циклу обходу агентів — порядок обходу впливав би на результат. Перший агент вже зайняв клітинку, другий не може. Перший вбив третього, бо обходив раніше. І так далі. Результат став би залежним від внутрішнього порядку, а не від декларативного `agent rule`.

Intent pattern розриває цей зв'язок. Усі агенти "пишуть лист дідусеві Морозу", а потім їх ілюзії розв'язуються одним великим арбітром. Семантика — чітка і визначена.

## Композиція: step = agent_phase ∘ cell_phase

Тепер найкрасивіше. Tick — це **композиція двох чистих функцій**:

```
step = agent_phase ∘ cell_phase
```

Де `∘` — математичний оператор композиції. У псевдокоді:

```python
def step(snapshot):
    return agent_phase(cell_phase(snapshot))
```

Береш вхідний snapshot, проганяєш через cell phase, отримуєш проміжний snapshot, проганяєш через agent phase, отримуєш фінальний snapshot. Усе. Це і є один tick.

А вся симуляція — це повторна композиція:

```
s_0 -- cell --> s_0' -- agent --> s_1 -- cell --> s_1' -- agent --> s_2 ...
```

Ви бачите, як гарно це лягає у функційну модель? Симуляція — це **не** "об'єкт, що мутує". Це **послідовність значень**, з'єднаних чистою функцією. Кожен snapshot можна заінспектувати, зберегти, переграти.

У OCaml це виглядало б ось так:

```ocaml
let step snapshot = snapshot |> cell_phase |> agent_phase

let run n initial_snapshot =
  let rec loop i snap acc =
    if i = 0 then List.rev (snap :: acc)
    else loop (i - 1) (step snap) (snap :: acc)
  in
  loop n initial_snapshot []
```

Уся історія симуляції — це просто `List<Snapshot>`. І вона існує **цілком**. Жодних "втрачених тіків", жодних "не знаю, що було на 846-му".

## Чому це важливо

Ця абстракція — не косметика. Вона відкриває чотири важливі властивості, кожна з яких важко або неможливо отримати в імперативній моделі.

### 1. Детермінізм

Якщо `tick : Snapshot -> Snapshot` — це чиста функція, то для одного й того самого вхідного snapshot вона завжди повертає той самий вихідний snapshot. Це і є визначення детермінізму.

У Swarmlet "та сама seed + та сама програма → ідентична послідовність snapshot-ів". Запустіть симуляцію двічі з однаковими параметрами — отримаєте біт-у-біт однакові виходи. Це гарантовано не "зусиллями інженерів", а **природою функції tick**.

Чому це важливо? Перевідтворюваність. Помилка, яка трапилася на 847-му тіку у Васі, трапиться на 847-му тіку у Петі — якщо вони запускають однакову програму з однаковою seed. Без цього distributed-debugging агентних симуляцій став би пеклом.

У Swarmlet є окремий тест-харнес ([SWARMLET-025](../../specification/SWARMLET_ISSUES.md)), що проганяє кожен референсний приклад двічі з однаковою seed і перевіряє, що snapshot-послідовності збігаються. Це безпосередня перевірка чистоти.

### 2. Тестування

Чисті функції — мрія для тестів. Підпис `cell_phase : Snapshot -> Snapshot` означає: "дай мені вхід, я скажу тобі вихід". Жодного setup, жодного mock-ування, жодних залежностей від часу або файлової системи.

Тест на forest-fire виглядає так:

```python
def test_burning_becomes_ash():
    snap_in = make_snapshot(states=[["Burning"]])
    snap_out = cell_phase(snap_in, program=forest_fire_program)
    assert snap_out.states[0][0] == "Ash"
```

Три рядки. Не треба об'єкта `World`, не треба запускати "цикл життя", не треба боротися з прихованим станом. Просто значення → функція → значення → assert.

Це і є те, що у FP-світі називають **referential transparency**. Ви можете замінити виклик функції її значенням, і нічого не зміниться. Бо у функції немає прихованого контексту.

### 3. Паралелізація

Я вже згадував, але повторю. У cell phase **усі клітинки незалежні**. Кожна обчислюється тільки з вхідного snapshot. Між ними немає причинно-наслідкового зв'язку у межах однієї фази.

Це означає: **вся cell phase ідеально паралелізується**. Розбийте сітку на 8 кускі, віддайте кожен кусок окремому потоку, зберіть результати — отримаєте те саме, що й у послідовному обході. Жодних блокувань, жодних races. Бо немає shared mutable state.

В імперативній моделі такого зробити **не можна**. Ви читаєте і пишете в один і той самий двовимірний масив, і будь-яке наївне розпаралелювання ламає семантику. Доводилося б вводити ту саму "стару + нова сітка" патерну — але то вже фактично було б рух у бік snapshot-моделі, тільки задом наперед.

Swarmlet v0.1 наразі не паралелізує (бо першочергова мета — простота і коректність), але **семантично шлях відкритий**. Будь-яка майбутня оптимізація може використовувати numpy vectorization, multiprocessing, або навіть GPU compute — і не зламає визначеної поведінки.

### 4. Time-travel debugging

Найкрутіша властивість. Якщо кожен tick породжує новий snapshot, а старий не мутується — ви можете **зберегти кожен snapshot** у пам'яті або на диску, і потім "перемотати" симуляцію назад.

```python
history = [initial_snapshot]
for _ in range(1000):
    history.append(step(history[-1]))
```

Тепер `history[847]` — це повний стан світу на 847-му тіку. Ви можете його відрендерити, проаналізувати, порівняти з `history[846]`, відновити з нього і запустити "альтернативну гілку" з іншою seed. Усе це — без жодних трюків, бо snapshot-и **просто є**.

В інтерпретаторі Swarmlet це безпосередньо використовується у форматі JSONL: кожен рядок вихідного файлу — це JSON-серіалізація одного snapshot-у. Ви можете завантажити будь-який рядок назад у пам'ять, і у вас знову повний світ на тому тіку.

```python
def to_json(self) -> str:
    return json.dumps(self.snapshot())
```

У імперативній моделі це коштувало б величезних зусиль. Тут — це безкоштовний наслідок архітектури.

## OCaml-міст: pipeline `|>` як зрозуміла нотація

Якщо синтаксис `agent_phase ∘ cell_phase` для вас занадто математичний, є більш програмістський спосіб написати те саме — pipeline operator. У OCaml (і F#, і Elixir) є оператор `|>`, що читається "потім".

```ocaml
let step snapshot =
  snapshot
  |> cell_phase
  |> agent_phase
```

Що це означає? "Візьми snapshot, передай його у cell_phase, потім результат — у agent_phase". Це той самий tick, але читається як **рецепт**, а не як математична формула.

Чому це краще? Бо порядок виконання збігається з порядком читання зверху вниз. Композиція функцій `agent_phase ∘ cell_phase` записується справа-наліво (спочатку cell, потім agent), що інтуїтивно дивно. А `|>` записує лівий-направо: "снепшот, потім cell, потім agent". Так, як говориться людською мовою.

Той самий прийом працює і у Python, тільки без синтаксичного цукру. Ось декілька еквівалентних форм:

```python
# Композиція
result = agent_phase(cell_phase(snap))

# Через проміжну змінну
intermediate = cell_phase(snap)
result = agent_phase(intermediate)

# Через reduce
from functools import reduce
phases = [cell_phase, agent_phase]
result = reduce(lambda s, f: f(s), phases, snap)
```

Усі три роблять те саме. Останній вигляд особливо цікавий — він явно представляє tick як "застосуй послідовність трансформацій до вхідного значення". Це і є pipeline-mindset у голому вигляді.

В Swarmlet програма теж читається як декларація pipeline:

```swl
let cell Tree =
  match neighborhood moore with
  | when has_neighbor Burning -> Burning
  | _ -> Tree

let agent Ant =
  let scent = field "pheromone" at here
  if scent > 0.5 then move forward
  else move (random_direction ())
```

Кожен `let` — це декларація однієї фази або одного rule. Композиція їх у tick — це робота інтерпретатора, але концептуально вона прозора: snapshot заходить, snapshot виходить, посередині — pipeline трансформацій.

Ця ментальна модель — ідеальна для дизайну агентних симуляцій. Замість "що мутувати в якому порядку" ви думаєте "які трансформації застосувати до світу". Це різниця між "як зробити" і "що зробити", класична декларативна перевага.

## Імперативний контраст: NetLogo-style mutate-in-place

Щоб картинка була повна, подивімося, як той самий tick виглядав би в імперативному стилі. Ось наївна реалізація, що **ламає** обидві властивості — і коректність, і паралелізованість:

```python
def tick_imperative(world):
    # Cell phase — мутуємо на місці!
    for y in range(world.height):
        for x in range(world.width):
            state = world.grid[y][x]
            if state == "Burning":
                world.grid[y][x] = "Ash"
            elif state == "Tree" and any_neighbor_burning(world.grid, x, y):
                world.grid[y][x] = "Burning"

    # Agent phase — теж мутуємо на місці!
    for agent in world.agents:
        if agent.energy > 10:
            target = pick_target(world.agents, agent)
            target.alive = False  # вбили
            agent.energy += 5
        agent.x, agent.y = move(agent)  # перемістили
```

Тут одразу два класи багів:

1. **Несинхронне оновлення клітинок.** `world.grid[y][x] = "Burning"` робиться в одній клітинці, і наступна клітинка вже бачить новий стан. Хвиля пожежі поширюється не дискретно, а вибухом.
2. **Залежність агентної фази від порядку обходу.** Перший агент вбив другого, і другий уже не діє. Третій агент мав би заспаунити нащадка у клітинку, але туди вже встиг переміститися перший. Усе крихке, усе залежить від `for ... in` порядку.

NetLogo, до речі, теж вирішує ці проблеми, просто інакше — через built-in примітиви типу `ask patches [...]` із прихованим snapshot-ом старого стану. Тобто концептуально NetLogo робить те саме, що Swarmlet, але обгортає це в imperative-looking синтаксис, що обманює очі. Swarmlet робить ту саму математику, але **визнає її явно**: snapshot — це сутність, tick — це функція. Жодного імперативного фасаду.

## Невеличкий чит: World.step() насправді мутує

Тепер чесна частина. Якщо ви уважно прочитали код у `engine.py`, то побачили: `World` — це **звичайний клас із стейтом**. У нього є `self.states`, `self.fields`, `self.agents`, і метод `step()` **мутує** ці атрибути.

```python
def step(self, n: int = 1):
    """Run n ticks."""
    for _ in range(n):
        self._cell_phase()
        self._agent_phase()
        self.tick_count += 1

def _cell_phase(self):
    next_states = self.states.copy()
    next_fields = {fn: arr.copy() for fn, arr in self.fields.items()}
    # ...
    self.states = next_states  # МУТАЦІЯ!
    self.fields = next_fields  # МУТАЦІЯ!
```

Що це? Невже ми обманювали увесь цей час?

Ні. Ось чому це чесно (і чому це **не** ламає функційну модель).

**Перше:** мутація відбувається на рівні **імплементації**, а не **семантики**. Концептуально — на вході у `_cell_phase` був snapshot S0, на виході — snapshot S1. Те, що ми переписали посилання `self.states` з масиву S0 на масив S1, — це деталь, як зберігати "поточний" snapshot у пам'яті.

**Друге:** ніхто ззовні не має посилання на старий масив `self.states`, поки `_cell_phase` працює. Усі обчислення йдуть із локальної змінної `next_states`, доки не закінчаться. Лише в кінці робиться swap. Це класична **copy-on-write** оптимізація — семантично immutable, фізично мутує під капотом.

**Третє:** snapshot **експортується** як immutable значення через метод `snapshot()`:

```python
def snapshot(self) -> Dict[str, Any]:
    return {
        "tick": self.tick_count,
        "states": state_grid,
        "fields": field_data,
        "agents": agents,
    }
```

Те, що ви отримуєте назовні — це чистий dict, не пов'язаний із внутрішніми буферами `World`. Ви можете його зберегти, серіалізувати, переграти — і він не зміниться, навіть якщо `World` далі рухається у нові тіки.

Це **performance-driven cheat**. Якби ми робили все буквально immutable у Python (з `frozendict` і `tuple` замість numpy arrays), симуляція 100×100 на 1000 тіків займала б години. З numpy + in-place буферами — секунди. Ціна — трохи дисциплінованості: не кешувати посилання на `self.states` поза рамками одного тіку.

Детальніше про межі чистоти і де ховаються побічні ефекти — у [purity-and-side-effects.md](08-purity-and-side-effects.md). Там ми обговорюємо, чому це нормально, доки ми чесні з собою щодо того, де мутація і яку інваріанту вона зберігає.

## Поширені пастки

Три речі, на які треба зважати, коли пишете код, що користується Swarmlet engine.

**Не зберігайте посилання на `world.states` між тіками.** Це numpy масив, і його ідентичність змінюється при кожному `step()`. Якщо ви зробили `my_ref = world.states` і викликали `world.step()`, ваш `my_ref` тепер **не пов'язаний** із поточним станом. Або беріть `world.snapshot()` (повна копія dict), або робіть `world.states.copy()` явно.

**Не пишіть у `world.fields[fname]` напряму ззовні.** Ви безпосередньо мутуєте внутрішній буфер інтерпретатора між тіками. Це порушить інваріанту, що snapshot[N+1] = step(snapshot[N]) — а від цієї інваріанти залежить детермінізм і time-travel debugging. Якщо вам треба ін'єктувати збурення — робіть це через cell rules або через initialization.

**Не плутайте порядок фаз.** `step()` робить cell phase, потім agent phase. Якщо ви пишете cell rule, що читає `agent_count_at(x, y)` — це число відображає **попередню** тікову позицію агентів, не поточну. Бо у момент cell phase агенти ще не рухалися на цьому тіку. Це не баг, це фіча: cell rule бачить "знімок до тіку". Але якщо ви очікуєте інше — здивуєтеся.

**Hint:** Для відновлення з JSON у dict — `json.loads`. Для diff не треба нічого складного: пройдіть по двовимірному масиву станів і виведіть координати, де `snap_n["states"][y][x] != snap_m["states"][y][x]`. Цей exercise показує, наскільки **дешево** робити time-travel debug, коли модель — функційна.

## Підсумок

Tick у Swarmlet — це не процедура, що мутує об'єкт. Це **чиста функція з підписом `Snapshot -> Snapshot`**, побудована як композиція двох інших чистих функцій: `cell_phase` і `agent_phase`. Усе складне (рух, конфлікти, kill, spawn) ховається всередині agent phase через intent pattern, що теж зберігає чистоту: спочатку збираємо intents як значення, потім застосовуємо атомарно.

Реалізація на Python внутрішньо мутує numpy буфери — це деталь продуктивності, що **не змінює семантики**. Зовнішнє API (метод `snapshot()`, серіалізація у JSON) повертає immutable значення.

З цього випливають чотири властивості: детермінізм без зусиль, тривіальна тестованість, готовність до паралелізації, і безкоштовний time-travel debugging. Жодну з них неможливо отримати у "справжньо імперативній" моделі без власноруч побудованого snapshot-механізму, що по суті був би тим самим, тільки задом наперед.

Якщо ви запам'ятаєте одну річ із цього документа — нехай це буде підпис: **`tick : Snapshot -> Snapshot`**. Все інше — слідство.
